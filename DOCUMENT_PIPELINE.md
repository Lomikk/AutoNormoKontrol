# Пайплайн вузовских документов под управлением ИИ

## Решение

Основной формат сдачи — PDF, собранный через LaTeX. Исходное содержание хранится
не в Word и не в свободном LaTeX, а в ограниченном Pandoc Markdown. Правила
оформления, их приоритет и происхождение хранятся отдельно. DOCX является
необязательным вторым backend и создаётся только тогда, когда его требует
преподаватель.

Это даёт четыре независимых слоя:

1. `content/` — смысл и структура документа;
2. `rules/` — нормализованные требования и их источники;
3. `render/latex/` — строгая реализация PDF-вёрстки;
4. `render/docx/` — необязательная реализация Word-вёрстки.

ИИ работает с текстовыми файлами и командами сборки. Пользователь не занимается
ручным форматированием ни в Word, ни в LaTeX.

## Почему именно так

### Почему не свободный LaTeX в главах

Свободный LaTeX позволяет добавлять локальные `\vspace`, `\fontsize`,
`\newgeometry` и другие хаки. Они быстро превращают документ в набор исключений,
который нельзя безопасно переоформить целиком.

Markdown оставляет в содержании только семантику: раздел, таблица, рисунок,
формула, ссылка, цитирование. Специальные элементы оформляются атрибутами и
именованными блоками, которые Pandoc-фильтр переводит в команды профиля.

### Почему не Quarto в первой версии

Quarto полезен для вычисляемых отчётов, notebooks и публикации в несколько
каналов, но для данного сценария добавляет ещё один конфигурационный слой.
Установленного Pandoc достаточно для преобразования Markdown в LaTeX и DOCX.
Quarto можно добавить позже, если появятся вычисляемые графики или сайт.

### Почему не автоматическая конвертация PDF/DOCX в LaTeX

Такая конвертация восстанавливает координаты и визуальные фрагменты, а не смысл
документа. Имеющиеся в рабочей папке результаты это подтверждают:

- `Яшков Андрей Михайлович-pdf2tex.com.tex` раскладывает страницу на сотни
  абсолютных фрагментов;
- `0000.tex` содержит повреждённую кодировку, изображения вместо элементов
  документа и даже координаты `NaN`/`Infinity`.

Оригинал DOCX/PDF следует использовать как измеряемый эталон, а не как исходный
код будущего шаблона.

### Зачем всё-таки нужен Word-слой

Если конечный файл обязан быть DOCX, правила должны быть представлены в
WordprocessingML хотя бы один раз. Это может быть автоматически сгенерированный
`reference.docx` и/или Open XML-постпроцессор. Ручное редактирование в интерфейсе
Word при этом не требуется.

Pandoc прямо предупреждает, что его промежуточное представление менее выразительно,
чем некоторые входные форматы, поэтому идеальная конвертация сложного LaTeX в DOCX
невозможна. Поэтому DOCX нужно собирать из общего семантического исходника, а не из
готового `.tex` или PDF.

## Модель правил

Каждое правило должно иметь не только значение, но и основание:

```yaml
tables:
  outer-border:
    value: dashed
    source: teacher-name
    evidence:
      file: evidence/teacher-comments.md
      location: "Комментарий от 2026-10-14"
    status: confirmed
    overrides: department-it
```

Рекомендуемый порядок приоритетов:

```text
университет
  < кафедра
  < тип работы
  < дисциплина
  < преподаватель
  < явно утверждённое исключение документа
```

Слияние не должно быть молчаливым. `resolve-rules.mjs` формирует:

- итоговый `resolved.yaml`;
- список конфликтов;
- объяснение, какое правило победило и почему;
- список правил без подтверждённого источника.

## Что генерировать автоматически, а что оставлять кодом

Из YAML автоматически генерируется типовая часть настроек:

- размер бумаги и поля;
- семейство, кегль и начертание шрифта;
- межстрочный интервал и абзацный отступ;
- базовые параметры заголовков;
- положение подписей;
- правила нумерации;
- выбранный библиографический стиль.

Сложные требования реализуются в `.sty` и Lua-фильтрах:

- нестандартные таблицы;
- особые титульные листы;
- ведомости и формы с фиксированной геометрией;
- необычная логика подписей и нумерации;
- условные элементы для конкретной кафедры.

Это практичный компромисс: типовые 80% синхронизируются между backend, а редкие
20% остаются обычным тестируемым кодом. Полный универсальный «компилятор всех
методичек» в первой версии не нужен.

## Контракт содержимого

В `content/*.md` разрешены:

- заголовки;
- абзацы, списки и сноски;
- таблицы и рисунки с идентификаторами;
- формулы;
- перекрёстные ссылки;
- библиографические ссылки;
- заранее определённые семантические блоки.

Запрещены:

- raw LaTeX и raw Open XML;
- ручные размеры шрифта и интервалы;
- принудительные вертикальные и горизонтальные отступы;
- ручная нумерация разделов, таблиц и рисунков;
- переносы страниц ради визуальной подгонки;
- оформление цветом или рамками напрямую в содержимом.

Если нужен локальный отступ или разрыв, это оформляется именованным правилом или
записью в `exceptions.yaml` с объяснением.

## Команды сборки

Пользователь и ИИ должны работать только через стабильные команды:

```powershell
./scripts/build.ps1 pdf
./scripts/build.ps1 docx
./scripts/test.ps1
```

PDF-сборка:

```text
validate rules
→ resolve profiles
→ validate semantic content
→ Pandoc Markdown to LaTeX
→ latexmk with LuaLaTeX
→ structural PDF checks
→ output PDF
```

DOCX-сборка:

```text
validate rules
→ resolve profiles
→ validate semantic content
→ Pandoc Markdown to DOCX
→ generated reference.docx styles
→ Open XML postprocessing
→ DOCX validation
```

Для LaTeX рекомендуется `latexmk -lualatex`: он сам запускает нужное число проходов
и вызывает Biber при изменении библиографии. `XeLaTeX` остаётся резервом для
пакетов, несовместимых с LuaLaTeX.

## Проверки

### Быстрые проверки каждой сборки

1. YAML соответствует схеме.
2. В содержимом нет запрещённых команд и raw-вставок.
3. Все рисунки и библиографические ключи существуют.
4. Все ссылки разрешены, нет `??`.
5. LaTeX завершается без ошибок.
6. Нет незапланированных переполнений строк и блоков.
7. PDF имеет A4, нужные шрифты и ожидаемое число страниц для fixtures.

### Регрессионные проверки профиля

После изменения оформления собираются небольшие fixture-документы. Их страницы
сравниваются с утверждёнными эталонами. Изменение таблиц не должно незаметно
сломать заголовки, формулы или список литературы.

`l3build` можно добавить для модульных тестов класса и пакетов. Для визуальной
вёрстки нужен отдельный рендер страниц PDF и image diff.

## Правила работы ИИ

При добавлении содержания ИИ:

1. изменяет только `content/` и библиографию;
2. не добавляет команды оформления;
3. запускает сборку и проверки;
4. сообщает о неоднозначной структуре, а не угадывает форматирование.

При изменении требования ИИ:

1. добавляет источник в `evidence/`;
2. изменяет соответствующий профиль `rules/`;
3. показывает конфликт с нижележащими правилами;
4. обновляет генератор или минимальный `.sty`/фильтр;
5. запускает все fixtures;
6. показывает визуально затронутые страницы;
7. не меняет содержание документа.

При фразе «у всех таблиц должна быть пунктирная рамка» изменение должно затронуть
профиль преподавателя и реализацию семантической таблицы, но не отдельные таблицы
в главах.

## DOCX: три режима качества

1. **Не нужен.** Канонический результат — PDF, Word-слой отсутствует.
2. **Нужен для комментариев.** Pandoc создаёт хорошо структурированный DOCX с
   базовыми стилями; совпадение переносов с PDF не обещается.
3. **Должен пройти строгий нормоконтроль в Word.** Используется утверждённый или
   программно сгенерированный `reference.docx`, Open XML-постпроцессор и проверка
   результата именно в Microsoft Word. Это отдельный backend, а не конвертация PDF.

Существующий `build-practice-materials.mjs` доказывает, что DOCX уже можно создавать
программно без интерфейса Word. Его проблема не в технологии, а в смешении текста,
данных и форматирования в одном большом файле. В новом пайплайне этот код стоит
сохранить только как прототип Word-backend для жёстких форм.

## План внедрения

### Этап 0 — завершить окружение

- дождаться завершения TeX Live;
- открыть новую PowerShell-сессию;
- проверить `lualatex --version`, `latexmk --version`, `biber --version`;
- проверить сборку минимального русского документа с Times New Roman;
- инициализировать Git перед началом рефакторинга.

### Этап 1 — PDF MVP

- взять один реальный нормоконтроль и один принятый образец;
- описать только подтверждённые правила;
- сделать один класс, один профиль и `everything.md`;
- собрать PDF через Pandoc, LuaLaTeX и latexmk;
- утвердить визуальный эталон.

### Этап 2 — управляемые overrides

- добавить deterministic merge профилей;
- создать отчёт конфликтов и provenance;
- добавить профиль конкретного преподавателя;
- проверить изменение на fixtures.

### Этап 3 — рабочий документ

- перенести содержание одной курсовой или отчёта;
- добавить библиографию, рисунки, таблицы и перекрёстные ссылки;
- запретить прямое форматирование автоматической проверкой;
- использовать этот проект в реальной сдаче.

### Этап 4 — DOCX только по факту требования

- определить, нужен ли редактируемый или строго оформленный DOCX;
- для простого режима использовать Pandoc;
- для строгого режима создать Word-backend из тех же правил;
- не пытаться добиться идентичности путём PDF-to-DOCX.

## Критерий успеха MVP

MVP готов, когда фраза «изменить отступ абзаца во всём документе» меняет одно
правило, перестраивает fixture и реальный документ, не затрагивает содержание и
создаёт объяснимый diff. DOCX не входит в критерий первой версии.
