## Что из этого следует

### 1. `Workspaces` — основное место хранения

На первом этапе поддерживаем только:

```text
AutoNormoKontrol/
├── Workspaces/
│   └── Моя курсовая/
└── ...
```

Возможность хранить работу где угодно считаем будущим удобством, но не требованием.

При этом содержимое работы не должно зависеть от `..\..\`. Относительный путь может использовать только маленький launcher. Если позднее появится запуск из произвольной папки, менять `content/`, метаданные и библиографию не потребуется.

Отказ от `PATH` был рациональным: он не решает необходимой сейчас задачи. Позднее его можно добавить как необязательное удобство.

### 2. Не нужно автоматически обеспечивать совместимость всех версий

Достаточно записывать в работу:

```yaml
created-with:
  engine-version: 0.1.0
  profile-id: susu-hsem-ceit-coursework-v1
  profile-digest: 166a859...
```

На другом компьютере движок проверяет это и сообщает:

```text
WARNING ANK-VERSION-MISMATCH

Работа создана AutoNormoKontrol 0.1.0.
Сейчас используется 0.2.0.

Draft можно продолжить, но результат может отличаться.
Для воспроизводимой сборки установите AutoNormoKontrol 0.1.0.
```

Самостоятельно искать, скачивать и запускать старую версию движок не обязан. Это действительно чрезмерно усложнило бы MVP.

### 3. Обновления не должны задавать вопросы во время сборки

Здесь я немного скорректирую пожелание.

Интерактивный вопрос во время `draft` плох для агентов и автоматизации:

```text
Обновить профиль? [Да/Нет]
```

Агент может зависнуть, случайно согласиться или не понять последствия.

Рациональнее:

- другая версия движка, но тот же профиль — предупреждение;
- изменился сам набор правил — сборка сообщает об этом;
- обновление выполняется отдельной явной командой.

Например:

```text
AutoNormoKontrol.cmd status
```

```text
Доступно обновление профиля:
susu-hsem-ceit-coursework-v1

Текущий digest: 166a859...
Новый digest:    820bc14...

Для просмотра:
AutoNormoKontrol.cmd profile-update --explain

Для применения:
AutoNormoKontrol.cmd profile-update
```

В MVP можно вообще не реализовывать обновление, а только предупреждать о несовпадении.

### 4. Требование преподавателя — не новый полный шаблон

Небольшая терминологическая поправка: это должен быть не обязательно новый LaTeX-шаблон, а новый профиль или дополнение к профилю.

Павел Константинович потребовал пунктирные таблицы:

```text
teacher-pavel-v1
└── переопределяет только оформление таблиц
```

Остальные требования продолжают поступать из базового профиля курсовой.

Именно здесь в будущем пригодится наследование. Но реализовывать его сейчас преждевременно. Пока нет реального второго профиля, мы не знаем:

- какие части действительно потребуется заменять;
- нужны ли несколько уровней;
- насколько часто преподаватели меняют требования;
- потребуется ли применять дополнение к уже существующей работе.

Первое реальное требование преподавателя станет хорошим основанием для минимального наследования. До этого текущий профиль остаётся самодостаточным.

Добавлять и публиковать такие профили действительно должен разработчик или продвинутый пользователь. Обычный пользователь не должен редактировать нормативную систему через интерфейс.

### 5. У работы один набор правил

Это сильно упрощает архитектуру:

```text
одна работа
→ один зафиксированный профиль
→ один канонический PDF
```

Не нужны:

- переключатели вариантов;
- несколько параллельных сборок;
- разные журналы аудита;
- разные content hashes;
- объяснение, какой PDF является основным.

Если когда-нибудь появится реальный случай двух вариантов, его можно будет рассмотреть отдельно.

### 6. Песочница остаётся контекстной

Техническая изоляция не требуется. Значит, нам не нужны сейчас:

- контейнеры;
- отдельные пользователи Windows;
- перехват shell-команд;
- собственный агентский шлюз;
- сложная система разрешений.

Достаточно, чтобы агент запускался в папке работы и видел понятные пользовательские файлы. Это снижает случайные ошибки, хотя не является защитой от злонамеренного процесса.

## Итоговый продуктовый контракт

На данный момент решения складываются так:

- AutoNormoKontrol хранится в одном переносимом каталоге;
- работы создаются внутри `Workspaces`;
- глобальный `PATH` не требуется;
- каждая работа переносима при наличии совместимого окружения;
- движок предупреждает о несовпадении версий, но не управляет старыми версиями автоматически;
- обновление правил не происходит во время Draft;
- одна работа использует один зафиксированный профиль;
- профили создаёт разработчик;
- наследование появится только вместе с первым реальным дополнением;
- агенты изолируются организацией папок, а не средствами безопасности ОС.

Это довольно небольшой и реалистичный объём.

## Что принимаем

### Создание проекта

При `new` пользователь указывает только:

- профиль работы;
- название папки.

Персональные данные остаются заполнителями. Позднее пользователь передаёт их агенту с явным разрешением.

Это уже соответствует текущему правилу: ИИ не выдумывает ФИО, группу, преподавателя, дисциплину и подписи.

### Пустой проект собирается сразу

Новая работа должна проходить Draft без предварительной анкеты:

```text
new
→ draft
→ PDF с заполнителями и предупреждениями
```

Дополнительная логика «документ слишком пустой» не нужна. Strict закономерно останется закрыт.

### Один постоянный PDF

Для предпросмотра:

```text
build/document.pdf
```

Для явного экспорта:

```text
output/document.pdf
```

Оба имени постоянные. При повторной сборке и экспорте файл заменяется атомарно. Историю версий пользователь хранит через Git или явный `archive`.

## Демонстрационное содержание нельзя помещать в настоящую работу

Идея показать агенту реальную нотацию хорошая. Но предложение «положить таблицу, формулу, рисунок, а агент потом удалит» опасно.

Агент может:

- забыть удалить демонстрационный объект;
- принять вымышленные данные за результаты исследования;
- оставить ссылку на несуществующий рисунок;
- отправить учебный текст в итоговую работу;
- удалить половину примера, нарушив структуру;
- потратить токены на очистку каждого нового проекта.

Это как раз тот случай, когда удобное на первый взгляд решение создаёт систематический риск.

Рациональнее разделить starter и примеры:

```text
Моя курсовая/
├── content/                 ← только настоящая работа
│   ├── 00-introduction.md
│   ├── 10-chapter-01.md
│   ├── 90-conclusion.md
│   └── 99-appendices.md
│
└── guide/
    └── notation-examples.md ← не участвует в сборке
```

В `notation-examples.md` можно показать:

- сноску;
- цитирование;
- таблицу;
- рисунок;
- график;
- формулу и блок «где»;
- приложение;
- первую и повторную ссылку.

Агент загружает файл только при необходимости. Он получает реальные примеры нотации, но учебные данные физически не могут попасть в PDF.

Starter при этом остаётся обезличенным:

```markdown
# Введение

<!-- CONTENT-TODO: актуальность, цель, задачи, объект, предмет, методы. -->
```

Пустое приложение лучше обозначить комментарием, а не создавать фиктивное «Приложение А»: СТО требует ссылку на каждое настоящее приложение.

## Пишущий агент не должен свободно менять движок

Здесь я должен прямо не согласиться.

Мощность агента не делает изменение движка безопасным. Если агент, запущенный для написания одной курсовой, изменит центральный фильтр, это может повлиять:

- на все остальные работы;
- на 172 требования;
- на Strict;
- на profile digest;
- на старые semantic review;
- на PDF других пользователей.

Небольшой override также может содержать Lua, TeX или PowerShell. Скачивание такого дополнения с GitHub фактически означает запуск чужого кода.

Правильное разделение такое:

### Простое изменение параметров

Например:

```text
интервал: 1,5
подпись таблицы: 12 pt
главы с новой страницы
```

В будущем можно разрешить как проверяемое декларативное дополнение без программного кода:

```yaml
teacher-override:
  id: teacher-pavel-v1
  evidence: "Замечание преподавателя от 20.09.2026"
  tables:
    outer-border: dashed
```

Движок принимает только известные поля и отклоняет всё остальное.

### Изменение поведения программы

Например:

- новый алгоритм нумерации;
- особая структура приложений;
- собственный Lua-фильтр;
- новая форма титульного листа.

Это уже разработка профиля. Она выполняется в отдельной сессии разработки:

```text
EngineRoot
→ изменение
→ тесты
→ check
→ Git
→ новая версия профиля
```

Пользователь может поручить это сильному агенту, но должен явно открыть ему репозиторий движка как разработчику. Пишущий агент из workspace не получает такое право автоматически.

Таким образом:

> Агент может быть и автором документа, и разработчиком движка, но не в одной неявной роли и не в одной рабочей папке.

## Как впоследствии будет выглядеть наследование

Целевая модель остаётся такой:

```text
СТО ЮУрГУ
  ↓
курсовая работа
  ↓
кафедра ЦЭИТ
  ↓
преподаватель Павел
```

Каждый следующий слой хранит только отличия.

Но в проект записывается уже зафиксированный результат:

```yaml

  digest: ...
```

Обычному агенту необязательно знать всю цепочку. Для диагностики пользователь сможет выполнить:

```text
AutoNormoKontrol.cmd rules --explain tables
```

и получить:

```text

teacher-pavel-v1
переопределяет:
susu-ceit-coursework-v1

Доказательство:
замечание преподавателя от 20.09.2026
```

Но реализовывать этот механизм до появления настоящего второго слоя всё ещё нерационально.

## Strict остаётся добровольным

Согласен: программа не должна запрещать пользователю забрать Draft.

Состояния следует честно различать:

```text

Strict не пройден
Смысловой аудит не завершён
Внешняя приёмка отсутствует
```

`export` может экспортировать последний успешный Draft, но должен предупредить:

```text
WARNING ANK-EXPORT-DRAFT

Экспортируется Draft.
AutoNormoKontrol не подтверждает полное прохождение нормоконтроля.

Ready: output/document.pdf
```

Strict — не разрешение пользоваться собственным документом, а максимальный уровень доказанной проверки.

Я бы не добавлял водяной знак «ЧЕРНОВИК» непосредственно в PDF: пользователь может осознанно захотеть распечатать или отправить Draft. Статус следует хранить в консоли и build-report.

## Слой — это область применения, а не одно правило

Все перечисленные пожелания одного преподавателя рациональнее хранить внутри одного дополнения:

```text
teacher-pavel-coursework-v1/
├── manifest.yaml
├── rules/
│   ├── tables.yaml
│   ├── equations.yaml
│   └── headings.yaml
├── styles/
└── tests/
```

Логическая цепочка:

```text
СТО ЮУрГУ
  ↓
курсовая работа
  ↓
кафедра ЦЭИТ
  ↓
требования преподавателя Павла
```

А уже последний слой содержит несколько независимых правил:

```yaml
overrides:
  - rule: tables.border
    value: dashed

  - rule: tables.font-size
    value: 11pt

  - rule: equations.spacing
    value: compact
```

Когда появляется новое пожелание преподавателя, создаётся новая версия того же дополнения:

```text
teacher-pavel-coursework-v1
→ teacher-pavel-coursework-v2
```

Люди обновляют один набор требований, а не собирают конструктор из двадцати микропакетов.

## Внутри дополнение может оставаться модульным

ИИ необязательно читать и менять весь профиль.

Пользователь говорит:

> Павел попросил уменьшить шрифт в таблицах.

Режим разработки предоставляет агенту только:

```text
manifest.yaml
rules/tables.yaml
tests/tables/
```

То есть внешне это один устанавливаемый набор, а внутри — небольшие тематические файлы.

Так сохраняются одновременно:

- удобная установка;
- маленький контекст агента;
- независимые тесты;
- понятная история Git;
- отсутствие десятков пакетов.

## Где разрабатывать пользовательские дополнения

Не в рабочей курсовой и не непосредственно в официальном движке.

Предлагаю будущую структуру:

```text
AutoNormoKontrol/
├── profiles/                 ← официальные профили
├── Extensions/               ← пользовательские дополнения
│   └── teacher-pavel-coursework-v1/
├── Workspaces/
└── AutoNormoKontrol.cmd
```

Сценарий:

```text
AutoNormoKontrol.cmd developer new-override
```

Пользователь выбирает:

```text
Базовый профиль:
Курсовая работа ЮУрГУ — ЦЭИТ

Название дополнения:
Требования преподавателя Павла
```

Создаётся отдельная папка разработки. Сильный агент запускается именно в ней:

```text
Extensions/teacher-pavel-coursework-v1/
```

Он получает:

- документацию формата дополнения;
- разрешённые параметры;
- тесты;
- команду проверки;
- базовый профиль только для чтения.

После успешной проверки дополнение можно:

- использовать локально;
- сделать отдельным Git-репозиторием;
- отправить другому студенту;
- позднее опубликовать в каталоге.

Это сохраняет ранее принятую границу: агент, пишущий курсовую, не редактирует движок. Но пользователь может открыть отдельную агентскую сессию в режиме разработчика.

## Простые и программные дополнения нужно различать

### Простые декларативные правила

Например:

- размер шрифта;
- интервал;
- положение подписи;
- начало главы с новой страницы;
- границы таблицы.

Они записываются как данные:

```yaml
tables:
  font-size: 11pt
  border: dashed
```

Такое дополнение относительно безопасно: движок принимает только известные поля.

### Программные дополнения

Например:

- новая форма титульного листа;
- нестандартный алгоритм нумерации;
- новый Lua-фильтр;
- произвольная LaTeX-реализация.

Такое дополнение содержит исполняемый код. Перед установкой нужны:

- явное предупреждение;
- проверка структуры;
- тесты;
- зафиксированный источник;
- точная версия и hash;
- ручное подтверждение пользователя.

Скачивание произвольного TeX/Lua с GitHub нельзя представлять как безобидную тему оформления.

## Как работа подключает дополнение

В будущем выбор может выглядеть так:

```yaml
profile:
  base: susu-hsem-ceit-coursework-v1
  overlays:
    - teacher-pavel-coursework-v2
```

А lock-файл фиксирует точный результат:

```json
{
  "base_profile": {
    "id": "susu-hsem-ceit-coursework-v1",
    "digest": "166a859..."
  },
  "overlays": [
    {
      "id": "teacher-pavel-coursework-v2",
      "digest": "93da10c..."
    }
  ],
  "effective_digest": "fe82b18..."
}
```

При сборке движок формирует один итоговый набор правил. Пишущему агенту не требуется разбираться в наследовании.

## Что происходит при конфликте

Молчаливое правило «последний файл победил» опасно.

Допустим:

```text
кафедра:
таблицы 12 pt

преподаватель:
таблицы 11 pt
```

Это нормальное переопределение: преподаватель находится на более конкретном уровне. Итог — 11 pt, а происхождение сохраняется.

Но если подключены два дополнения одного уровня:

```text
teacher-pavel-a: таблицы 11 pt
teacher-pavel-b: таблицы 10 pt
```

движок должен остановиться:

```text
ERROR ANK-PROFILE-CONFLICT

Два дополнения одного уровня задают разные значения:

tables.font-size:
  teacher-pavel-a → 11pt
  teacher-pavel-b → 10pt

Выберите одно дополнение или создайте явное решение конфликта.
```

Никакого случайного выбора по алфавиту или порядку файлов.

## Если маленьких исправлений на GitHub много

Рациональная модель публикации:

```text
один преподаватель + один тип работы + устойчивый контекст
→ один репозиторий дополнения
```

Например:

```text
ank-overrides/
├── susu-ceit-pavel-coursework
├── susu-ceit-pavel-practice-report
└── susu-ceit-ivanov-coursework
```

Внутри одного репозитория могут накапливаться десятки правил и версий.

Если требования преподавателя различаются по дисциплинам, тогда имеет смысл отдельный набор:

```text
teacher-pavel-databases-coursework-v1
teacher-pavel-economics-coursework-v1
```

Но разделять нужно только после появления реального различия, не заранее.

## Что фиксируем, но пока не реализуем

Сейчас достаточно не зацементировать следующие границы:

- профиль разрешается через одну функцию движка;
- работа хранит ID и digest итогового профиля;
- формат manifest имеет версию;
- workspace не указывает произвольные исполняемые файлы;
- официальные профили и пользовательские дополнения физически разделены;
- обновления применяются только явно;
- один устанавливаемый набор может содержать много тематических правил.

Полноценные `Extensions`, GitHub-установка и наследование начнём после появления первого реального дополнения. До этого они остаются спроектированным направлением, а не кодом «на всякий случай».
