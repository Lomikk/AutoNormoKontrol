Да, мысли логичные. Более того, это уже складывается в **простую и устойчивую модель продукта**:

```text
Движок AutoNormoKontrol
├── хранит профили и шаблоны работ
├── создаёт новые проекты
├── знает, как проверять и собирать каждый профиль
└── обновляется отдельно

Проект работы
├── содержит только сам документ
├── является рабочей областью пользователя и агента
├── не содержит реализацию движка
└── обращается к движку через маленький локальный launcher
```

Это лучше текущего состояния, где агент видит одновременно курсовую, PowerShell-скрипты, схемы, нормативные источники и внутренности конвертации.

## Как это должно выглядеть

### Центральный движок

```text
AutoNormoKontrol/
├── AutoNormoKontrol.cmd
├── profiles/
│   ├── susu-coursework-v1/
│   ├── susu-thesis-v1/
│   └── susu-practice-report-v1/
├── scripts/
├── schemas/
├── sources/
└── ...
```

Пользователь запускает:

```text
AutoNormoKontrol.cmd
```

и получает сценарий:

```text
1. Создать новую работу
2. Открыть существующую работу
3. Показать доступные шаблоны
4. Проверить установку
5. Выход
```

При создании:

```text
Выберите тип работы:

1. Курсовая работа
2. Выпускная квалификационная работа
3. Отчёт по практике
```

После выбора движок создаёт отдельную папку.

### Рабочая песочница

```text
Моя курсовая/
├── AutoNormoKontrol.cmd
├── project.yaml
├── GEMINI.md
├── metadata.yaml
├── bibliography.bib
├── assets-manifest.json
├── content/
├── assets/
├── research/
└── build/
```

Именно в этой папке запускаются:

- Gemini CLI;
- Aider;
- OpenCode;
- Codex;
- обычный редактор пользователя.

Агент видит только конкретную работу.

## Локальный `AutoNormoKontrol.cmd` не должен содержать движок

Это должен быть **тонкий посредник**, условно:

```bat
@echo off
call "%AUTONORMOCONTROL_HOME%\AutoNormoKontrol.cmd" --workspace "%~dp0" %*
```

То есть команда:

```text
AutoNormoKontrol.cmd draft --quiet
```

фактически превращается во внутренний вызов:

```text
Центральный движок:
собери workspace C:\...\Моя курсовая
по профилю susu-coursework-v1
в режиме draft quiet
```

Локальный файл не знает ничего о Pandoc, TeX, схемах и фильтрах. Он только передаёт:

- путь рабочей папки;
- название команды;
- аргументы.

Это правильная граница.

## Какие команды нужны внутри песочницы

Агенту, занимающемуся документом, достаточно небольшого интерфейса:

```text
AutoNormoKontrol.cmd status
AutoNormoKontrol.cmd check
AutoNormoKontrol.cmd draft
AutoNormoKontrol.cmd draft --quiet
AutoNormoKontrol.cmd strict
AutoNormoKontrol.cmd export
AutoNormoKontrol.cmd doctor
```

Возможно ещё:

```text
AutoNormoKontrol.cmd rules
```

чтобы получить краткое описание требований текущего профиля.

Ему не нужны команды:

```text
profiles add
profile develop
schemas validate-internal
sources migrate
engine test
template publish
```

Они остаются только в центральном движке.

## `project.yaml` фиксирует выбранный профиль

В каждом созданном проекте нужен маленький манифест:

```yaml
project-version: 1
profile:
  id: susu-hsem-ceit-coursework-v1
  version: 1
engine:
  minimum-version: 1.0.0
document:
  type: coursework
```

Тогда локальная команда ничего не угадывает. Движок видит:

> Эту папку надо собирать как курсовую по профилю `susu-hsem-ceit-coursework-v1`.

Позже туда можно добавить хеш профиля или lock-файл для воспроизводимости:

```yaml
profile:
  id: susu-hsem-ceit-coursework-v1
  version: 1
  content-hash: ...
```

Это защитит старую работу от неожиданного изменения после обновления профиля.

## Важное уточнение про «смену шаблона»

Здесь лучше разделить две операции.

### Нужна сейчас

**Выбор шаблона при создании новой работы:**

```text
new
→ выбрать курсовую
→ создать отдельную рабочую папку
```

### Не нужна в первом MVP

**Смена шаблона у уже существующей работы:**

```text
курсовая
→ внезапно заменить профиль на диплом
```

Это сложная миграция:

- другая структура;
- другие обязательные разделы;
- другие метаданные;
- другие формы;
- другие проверки.

Поэтому профиль проекта после создания лучше считать зафиксированным.

Позже можно сделать отдельную осознанную команду:

```text
AutoNormoKontrol.cmd migrate-profile
```

но не обычный переключатель.

Иначе «смена шаблона» быстро превратится в источник повреждённых проектов.

## Тихая сборка становится частью контракта

Для агента:

```text
AutoNormoKontrol.cmd draft --quiet
```

Успех:

```text
OK
PDF: build/document.pdf
```

Ошибка:

```text
FAILED: 2 errors

ERROR [REQUIRED_FIELD]
metadata.yaml: supervisor.name
Поле руководителя не заполнено.

ERROR [MISSING_CITATION]
content/01-literature-review.md:42
Источник `ivanov2024` отсутствует в bibliography.bib.

Full log: build/logs/draft.log
```

Полный технический вывод движок сохраняет в файл, но не помещает в контекст агента.

Это полезно не только Gemini, а вообще для всех клиентов:

- Codex;
- Aider;
- OpenCode;
- CI;
- будущий GUI.

## Насколько песочница действительно ограничивает агента

Разделение папок отлично решает две задачи:

- агент не получает технический шум в контекст;
- агенту сложнее случайно изменить движок.

Но это **контекстная граница, а не настоящая граница безопасности**.

Например, Gemini CLI с надписью:

```text
no sandbox
```

через shell теоретически может обратиться к файлам вне проекта. Чтобы действительно запретить выход, позже понадобятся:

- tool policies;
- sandbox;
- разрешённые команды;
- запрет произвольного shell;
- либо OpenCode-прослойка с узкими инструментами.

Для обычной работы с доверенной моделью разделения каталогов достаточно. Для жёсткой безопасности — нет.

## Что осталось до цельного MVP

По сути, перечень уже небольшой:

1. **Каталог профилей**  
   Команды просмотра доступных типов работ.

2. **Создание проекта**  
   Выбор профиля, имени и пути; копирование начальной структуры.

3. **`project.yaml` и фиксация профиля**  
   Чтобы движок однозначно понимал, как собирать проект.

4. **Локальный обрезанный `AutoNormoKontrol.cmd`**  
   Только прокси к центральному движку.

5. **Сборка по пути workspace**  
   Центральный движок должен принимать не только текущий каталог, но и явно переданный проект.

6. **`draft --quiet`**  
   Краткие ошибки в консоли, полный лог в файле.

7. **Чистая агентская песочница**  
   `content`, `metadata`, `bibliography`, `assets`, `research`, инструкции агента.

8. **`status` и `export`**  
   Чтобы пользователь мог закончить полный жизненный цикл документа.

AI-прослойка после этого действительно перестаёт быть срочной. У каждого агента будет одинаковый понятный контракт:

```text
Вот рабочая папка одного документа.
Вот файлы, которые можно менять.
Вот команда проверки.
Вот компактный результат.
Внутренности движка тебя не касаются.
```

Это логичная архитектура и для человека, и для агента. Главное — не делать локальный `AutoNormoKontrol.cmd` второй копией движка: он должен остаться маленьким мостом к единственной центральной реализации.
