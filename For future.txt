Да — **OpenCode с программной прослойкой потенциально позволит расходовать токены заметно экономнее**. Но экономию даст не сам OpenCode, а возможность жёстко определить:

- какой системный контекст отправляется;
- какие инструменты видит модель;
- какие файлы читаются;
- что возвращается после выполнения инструмента;
- когда история сжимается или выбрасывается.

Если запустить OpenCode «как есть», положить туда тот же 500-строчный промпт и открыть весь проект, большой разницы с Gemini CLI может не быть.

## Почему «два запроса» потратили 269 тысяч токенов

По вашему скриншоту Flash Lite использовал:

```text
47 / 500 RPD
269 040 / 250 000 TPM
```

Значит, это были не два API-запроса. Это были два **пользовательских сообщения**, которые породили множество внутренних обращений модели:

```text
запрос пользователя
→ модель решает прочитать файл
→ результат файла возвращается модели
→ модель читает другой файл
→ результат снова возвращается модели
→ модель читает PDF
→ модель формирует ответ
→ пользователь разрешает изменение
→ модель читает раздел
→ модель редактирует
→ модель проверяет результат
```

Gemini CLI действительно сохраняет в истории все вызовы инструментов и их результаты, а его программный вывод отдельно показывает события `tool_use`, `tool_result` и итоговую статистику токенов. 

Каждый такой шаг может снова нести:

- встроенный системный промпт Gemini CLI;
- `GEMINI.md`;
- 500 строк `SYSTEM_PROMPT`;
- описания доступных инструментов;
- предыдущий разговор;
- результаты чтения файлов;
- извлечённый PDF;
- вывод shell-команд.

Именно это надо оптимизировать.

# Чем здесь поможет OpenCode

OpenCode имеет SDK и серверный API, через которые можно программно управлять сессиями и агентом. Он также позволяет создавать собственные инструменты и настраивать автоматическое сжатие контекста с удалением старых результатов инструментов. 

Вместо универсальных инструментов:

```text
read
write
grep
glob
bash
webfetch
```

можно дать модели только пять узких:

```text
read_document_section
update_document_section
inspect_source
update_bibliography
build_draft_quiet
```

Например, `build_draft_quiet` возвращает модели не тысячу строк, а:

```text
FAILED

ERROR [CITATION_MISSING]
content/02-main.md:48
Citation `macfarlane2026` is absent from bibliography.bib.

Full log: build/logs/draft.log
```

А `inspect_source` не отправляет весь 300-страничный PDF в основной разговор, а возвращает:

```json
{
  "title": "Pandoc User's Guide",
  "author": "John MacFarlane",
  "document_date": "2026-06-03",
  "relevant_sections": [
    "Pandoc architecture",
    "Markdown syntax",
    "citations"
  ],
  "local_path": "research/originals/MANUAL.pdf"
}
```

Полный текст открывается только по отдельному запросу модели.

Это и есть реальная экономия.

## Ещё одно преимущество OpenCode

Большой профиль можно разделить на загружаемые по необходимости навыки:

```text
core
bibliography
figures
tables
formulas
metadata
semantic-review
final-acceptance
```

OpenCode показывает агенту только название и краткое описание навыка, а полное содержимое `SKILL.md` загружает, когда навык реально понадобился. 

Например, при заполнении ФИО модели не нужны:

- правила формул;
- правила приложений;
- оформление библиографии;
- финальный аудит;
- требования к таблицам;
- внешняя приёмка.

Сейчас всё это каждый раз находится в `SYSTEM_PROMPT_SUSU_COURSEWORK.md`.

# Но сначала можно сильно облегчить Gemini CLI

Переходить на OpenCode немедленно необязательно. Gemini CLI уже имеет почти все необходимые точки оптимизации.

## 1. Сократить постоянный промпт

В постоянной части должны остаться только правила, которые применяются **к каждому действию**:

```markdown
# Роль

Помогай писать и проверять вузовскую работу.

# Постоянные запреты

- Не выдумывай источники, авторов, DOI, страницы и результаты.
- Не изменяй файлы без явного согласия пользователя.
- Не изменяй движок и профиль.
- Используй только семантический Markdown.
- После правок проверяй документ тихой сборкой.
- При аналитическом вопросе сначала отвечай, не изменяя файлы.

# Рабочая область

- metadata.yaml
- content/**
- bibliography.bib
- assets/**
- research/**

# Проверка

Команда: AutoNormoKontrol.cmd draft --quiet
```

А специфические правила вынести отдельно:

```text
instructions/
├── bibliography.md
├── figures.md
├── tables.md
├── formulas.md
├── appendices.md
└── final-review.md
```

Модель читает нужный файл только при соответствующей задаче.

`semantic-review.yaml` и `external-acceptance.yaml` вообще не должны попадать в обычную писательскую сессию.

## 2. Ограничить вывод shell

Gemini CLI поддерживает автоматическое суммирование вывода `run_shell_command` с заданным токенным бюджетом. Также можно раньше включить компрессию контекста. 

В проектном `.gemini/settings.json`:

```json
{
  "model": {
    "name": "gemini-3.1-flash-lite",
    "summarizeToolOutput": {
      "run_shell_command": {
        "tokenBudget": 1000
      }
    },
    "compressionThreshold": 0.25
  }
}
```

Но собственный `draft --quiet` лучше автоматического пересказа: движок точно знает, какие строки являются ошибками, а модель-суммаризатор может пропустить важную деталь.

## 3. Сжимать завершённые задачи

После законченной операции:

```text
/compress
```

Gemini CLI заменяет всю историю кратким резюме, сокращая последующие входные запросы. 

Практический ритм:

```text
добавили источник
→ проверили Draft
→ /compress

доработали введение
→ проверили Draft
→ /compress
```

Не стоит вести всю курсовую в одном бесконечном несжатом контексте.

## 4. Проверить работу кэша

При API-key-аутентификации Gemini CLI автоматически использует кэширование повторяющегося контекста. Статистику можно посмотреть через:

```text
/stats
```

Там отображаются обычные и кэшированные токены. 

Но я бы не рассчитывал на кэш как на основное решение: документация Gemini определяет TPM как число входных токенов в минуту, а превышение любого лимита приводит к `429`. Из публичной документации не вполне ясно, как именно кэшированные токены учитываются в вашей конкретной квоте, поэтому ориентироваться нужно на фактический `/stats` и AI Studio. 

## 5. Можно скрывать результаты инструментов

Gemini CLI поддерживает hook `AfterTool`, который может обработать результат инструмента или вообще не помещать его в контекст модели. 

Например:

```text
AutoNormoKontrol вывел 1000 строк
→ пользователь при желании видит полный лог
→ hook извлекает ошибки
→ модель получает 20 строк
```

То есть тихую сборку можно реализовать двумя способами:

1. нативно в AutoNormoKontrol — предпочтительно;
2. временно через Gemini CLI hook.

# Стоит ли строить прослойку на OpenCode

**Да, если мы решим делать собственный пользовательский режим AutoNormoKontrol AI.**

Тогда OpenCode подходит как скрытый runtime:

```text
Пользователь
↓
простое меню или чат AutoNormoKontrol
↓
наша прослойка
↓
OpenCode SDK
↓
Gemini API
```

Прослойка будет отвечать за:

```text
выбор текущего раздела;
минимальный системный промпт;
дозированную загрузку правил;
компактные инструменты;
тихую сборку;
состояние документа;
сжатие истории;
маршрутизацию моделей.
```

Но для текущего этапа я бы не начинал писать её. Сначала разумнее:

1. разделить workspace и движок;
2. реализовать `draft --quiet`;
3. сократить основной промпт;
4. разнести редкие правила по тематическим файлам;
5. включить суммирование shell-вывода;
6. посмотреть `/stats` после одной одинаковой задачи.

После этого повторить тест. Если Flash Lite всё ещё съедает TPM за пару пользовательских действий, тогда OpenCode с пятью собственными инструментами будет уже обоснованным следующим шагом, а не очередной архитектурой «на всякий случай».

Итог:

> **OpenCode способен дать более строгий контроль над токенами, но большую часть текущей проблемы создаёт не Gemini CLI как таковой, а огромный постоянный промпт, широкие инструменты и многократно возвращаемые результаты. Сначала нужно устранить их.**


Так, с агентами понятно, у нас уже есть конкретные идеи оптимизации и работы с ними.
Aider — самый нищанский, работа через браузер.
Google — почти то же самое, что и Codex, но требуется создать отдельную папку-песочницу для него, чтобы не лез в движок.
Open Code потенциально дешевле, чем Google, но требует технической реализации.

Для всех нужно сделать тихую сборку и оптимизировать промты, ибо сейчас у нас реально много текста read.me/AGENTS.md системного промта, и ещё бог знает, что они там читают.
