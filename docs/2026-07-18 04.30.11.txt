Проведи небольшую доработку AutoNormoKontrol по результатам реального
использования профиля статьи. Не выполняй крупный рефакторинг и не меняй
архитектуру без необходимости.

1. Обработка ошибок latexmk

Была ситуация:

- первая сборка завершилась кодом 12;
- в журнале latexmk сообщал о предыдущей ошибке;
- article.pdf отсутствовал;
- повторная сборка позже успешно прошла без очистки build.

Нельзя автоматически утверждать, что причина — блокировка PDF или повреждённый
кэш latexmk.

Добавь более надёжную обработку:

- если latexmk сообщает, что цель актуальна, но build/article.pdf отсутствует,
  автоматически выполнить принудительную пересборку;
- сначала использовать безопасный повторный запуск с принудительным rebuild;
- не требовать от пользователя удалять весь каталог build;
- при повторной ошибке вывести последние значимые строки article.log;
- различать:
  a) файл занят другим процессом;
  b) отсутствующий PDF при состоянии up-to-date;
  c) реальную ошибку LaTeX;
  d) ошибку Biber;
  e) ошибку сборки ассета.

Не удаляй asset-кэш без необходимости.

2. Повторная сборка ассетов

Сейчас при каждой сборке заново компилируются все объекты из manifest,
включая неиспользуемые в текущей статье.

Нужно:

- не пересобирать ассет, если его исходники не изменились и выходной PDF существует;
- по возможности собирать только ассеты, на которые ссылается текущий Markdown;
- не собирать старую схему, отсутствующую в статье;
- в отчёте кратко показывать:
  built / cached / skipped.

3. Удобство вывода

Текущий журнал слишком длинный: пользователь видит сотни строк загрузки
пакетов LaTeX.

В обычном режиме показывать краткий вывод:

- проверка Markdown;
- список ассетов и их состояние;
- Pandoc;
- LuaLaTeX;
- Biber;
- число страниц;
- путь к PDF.

Полный журнал сохранять в build, а в консоль выводить его только при ошибке
или при включённом verbose-режиме.

4. Состояние проекта

Исправить вывод:

“PDF postflight: passed ( стр.)”

Количество страниц не должно быть пустым.

Если PDF отсутствует, нельзя одновременно показывать postflight passed без
пояснения. Нужно различать:

- postflight текущего PDF;
- результат предыдущей сборки;
- PDF отсутствует.

5. Работа с несколькими заданиями в одном workspace

Сейчас замена article.md и metadata.yaml фактически уничтожает предыдущую
работу или требует ручных резервных копий.

Добавь простой сценарий:

- создать вариант документа или снимок перед заменой;
- дать понятные команды «сохранить текущую версию» и «начать новую статью»;
- не создавать новый профиль: это две статьи одного типа;
- сохранять bibliography и подходящие ассеты, но не переносить неиспользуемые
  схемы автоматически.

6. Проверка требований задания

Если пользователь указал объём в страницах, структура задания должна считаться
критерием приёмки.

После сборки сообщать:

- фактическое число страниц;
- требуемый диапазон;
- соответствует / не соответствует.

Успешная техническая сборка не означает выполнение задания.

7. Честный agent-report

Запретить утверждения, не подтверждённые выполненными действиями:

- «PDF визуально проверен», если страницы не открывались или не анализировались;
- «применялся OCR», если PDF только читался;
- «соответствует СТО», если конкретный стандарт не задан;
- «все факты подтверждены», если не выполнена проверка соответствия утверждений
  источникам.

Разделять:
- техническую сборку;
- автоматический postflight;
- визуальную проверку;
- содержательную проверку человеком.

8. Уровень сложности текста

Агент должен учитывать уровень автора и характер задания.

Для обычной бакалаврской практики:
- не превращать простой текст в исследование уровня магистратуры;
- избегать искусственно сложных моделей, терминологии и громких выводов;
- выбирать понятную тему и практические рекомендации;
- не использовать старую схему только потому, что она уже существует;
- для новой статьи создавать новую иллюстрацию, соответствующую её содержанию.

Сделай минимальные изменения, добавь тесты для новых сценариев и представь
отчёт:
- какие файлы изменены;
- какие проблемы устранены;
- какие ограничения остались;
- как воспроизвести проверку.
