# Контракт требований v2

Статус: действующий контракт движка начиная с profile schema v2.

Эта нотация устраняет три вручную синхронизируемых списка: canonical IDs,
review inventory и шаблоны журналов. Источниками истины остаются два файла:

1. нейтральный `inventory.json` рядом с нормативным источником;
2. `requirements.json` внутри конкретного полного профиля.

Generated-файлы в `build/` и журналы новой работы являются производными. Они не
могут незаметно изменить применимость нормативного пункта.

## 1. Граница ответственности

`inventory.json` отвечает только на вопрос: **что существует в источнике**.
Он содержит стабильный ID, тип записи, нейтральное резюме и точный locator.
Он не знает кафедру, тип работы, LuaLaTeX, способ проверки и текст ошибки.

```json
{
  "id": "STO-8.5.4",
  "kind": "requirement",
  "summary": "На каждый рисунок в тексте должна существовать ссылка",
  "source": {
    "document_id": "STO-21-2008",
    "clause": "8.5.4",
    "locator": "sources/susu/sto-21-2008/derived/standard.layout.txt"
  }
}
```

Профильный `requirements.json` отвечает на вопрос: **как этот пункт действует
в данном профиле**. Один и тот же source inventory позднее может использоваться
несколькими независимыми профилями без скрытого наследования между ними.

Локальная обязанность, которой нет в нормативном inventory, не получает
поддельный `STO-*`. Вместо `source_ref` она объявляет собственный ID, `summary`
и происхождение:

```json
{
  "id": "TEACHER-PAVEL-TABLE-BORDER",
  "origin": {
    "kind": "teacher",
    "source": "Письменное замечание преподавателя",
    "locator": "research/teacher-feedback-2026-07-19.md"
  },
  "summary": "Внешняя граница таблиц должна быть пунктирной"
}
```

Такой пункт участвует в тех же programmatic/semantic/external gates, но не
меняет exact-set исходного СТО. `source_ref` и `origin` взаимоисключаемы.

## 2. Disposition

Каждый пункт source inventory обязан ровно один раз присутствовать в полном
профиле и получить одно значение:

- `applicable` — обязанность действует;
- `conditional` — действует при явно описанном условии;
- `formal` — определение, групповой заголовок, пример или иная запись, которую
  нужно сохранить в exact-set, но нельзя объявлять отдельной проверяемой
  обязанностью;
- `not-applicable` — пункт достоверно не применяется к этому профилю.

`formal` не означает «реализовано другим пунктом». Если источник содержит
обязанность, её нельзя скрыть этим статусом: нужно объявить реальный
verification либо `not-applicable` с объяснением в `notes`.

## 3. Verification

У применимого или условного пункта есть один или несколько способов проверки.
Смешанный пункт разрешён: например, геометрию можно проверить кодом, а решение
кафедры о допустимом backend получить только извне.

### Programmatic

```json
{
  "kind": "programmatic",
  "check": "document.element-order",
  "parameters": {
    "first": "conclusion",
    "then": "bibliography"
  },
  "diagnostic": "STO-7.11.2/ELEMENT_ORDER",
  "severity": "error"
}
```

`check` выбирается только из allow-list движка. Значение не является именем
скрипта, shell-командой, regex или путём к исполняемому файлу. Неизвестное
значение отклоняется fail-closed.

`profile.handler` — явный переходный тип для старой проверки, которая пока
остаётся в профильном Lua/TeX/postflight-коде. Он не выдаётся за универсальный
алгоритм. По мере миграции такие записи заменяются конкретными checks.

Programmatic-пункт содержит `coverage.implementation_markers` и
`coverage.test_markers`. Реальный обработчик и реальный тест обязаны содержать
комментарий с ID пункта.

### Semantic

```json
{
  "kind": "semantic",
  "review_id": "STO-5.4",
  "question": "Текст написан научным языком, а термины единообразны",
  "evidence_required": true
}
```

Из записи автоматически появляется строка в новом
`compliance/semantic-review.yaml`. Генератор устанавливает только
`status: uncertain` и пустое evidence. Он никогда не присваивает `pass` и не
переносит результат между снимками содержания.

### External

```json
{
  "kind": "external",
  "review_id": "STO-8.1.3/backend",
  "required_evidence": "Решение кафедры о допустимости LuaLaTeX"
}
```

Из записи появляется item в `external-acceptance.yaml`. Генератор не может
принять решение за кафедру, преподавателя или пользователя.

## 4. Диагностика

Статический текст ошибки и подсказка принадлежат требованию:

```json
"diagnostics": {
  "REFERENCE_MISSING": {
    "message": "На рисунок отсутствует ссылка в тексте",
    "hint": "Добавьте первое упоминание перед рисунком"
  }
}
```

Обработчик выдаёт код и динамические параметры события. Код обязан иметь вид
`<requirement-id>/<LOCAL_CODE>` и ссылаться на существующую запись. Номер
страницы, имя файла, строка, ID рисунка и фактическое значение не дублируются в
реестре: они добавляются обработчиком как детали конкретной ошибки.

Технические ошибки движка не маскируются под СТО. Для них развивается отдельный
каталог `ENGINE-*` в рамках `R1.4b/diagnostics-v1`.

## 5. Структура документа

Структура не хранится одним вымышленным «общим пунктом». Каждый реальный пункт
добавляет собственное ограничение:

- `document.required-elements` — обязательные семантические элементы;
- `document.element-order` — одно отношение `first → then`;
- `document.visible-element` — ожидаемый печатный элемент PDF.

Компилятор объединяет отношения в граф и отклоняет цикл. Поэтому правило
`assignment → abstract` может происходить из одного пункта, а
`conclusion → bibliography` — из другого.

`project.yaml: document.content` по-прежнему является точным списком и порядком
Markdown-файлов конкретной работы. Главу можно дробить на любое число файлов:
проверка выполняется над объединённым Pandoc AST и семантическими элементами, а
не над именем `02-main.md`.

`requirements.json` не генерирует `profile.yaml`, LaTeX-шаблон или правильный
текст документа. Он независимо объявляет и проверяет их контракт. Это не даёт
ошибке в одном генераторе одновременно создать неверный результат и объявить
его правильным.

## 6. Производные артефакты

При `new` движок один раз создаёт:

- `compliance/semantic-review.yaml`;
- `compliance/external-acceptance.yaml`.

Существующие журналы не перезаписываются: в них находятся решения конкретной
работы.

При каждой сборке создаются:

- `build/effective-requirements.json` — объединённый машинный контракт;
- `build/effective-requirements.yaml` — metadata для Pandoc/Lua;
- traceability-отчёты с hash source inventory и profile requirements.

Оба исходных файла входят в `profile_digest`. Изменение системного prompt
остаётся предупреждением совместимости workspace, а не запретом правки
локальной копии prompt.

## 7. Добавление или изменение пункта

1. Если изменился нормативный корпус, сначала изменить source inventory и
   locator; профиль не может сам придумать новый source clause.
2. Обновить ровно одну запись профильного `requirements.json`.
3. Для programmatic-проверки выбрать существующий allow-listed check либо
   реализовать новый общий алгоритм без исполнения данных из JSON.
4. Добавить diagnostic message/hint, обработчик и тест с тем же ID.
5. Выполнить `./AutoNormoKontrol.cmd check`.

Удаление записи только из `requirements.json`, неизвестный check, неизвестная
diagnostic-ссылка, дубликат review ID и цикл порядка завершаются ошибкой.

## 8. Текущая граница миграции

Schema v2 уже является единственной поддерживаемой схемой профиля. Активный
coursework-профиль содержит 172 source entries, 60 semantic review items и 25
external acceptance items. Декларативно перенесены базовые обязательные
элементы, before/after constraints и видимая последовательность PDF.

Остальные старые programmatic-проверки пока честно помечены
`check: profile.handler`; их диагностические строки переносятся из обработчиков
постепенно, по мере добавления regression fixtures. Это ограничивает diff и не
ослабляет уже работающий coverage gate.
