# Roadmap AutoNormoKontrol

Статус документа: живая архитектурная дорожная карта. Она фиксирует допустимое
направление развития, но не объявляет универсализацию текущим приоритетом.
Текущий продукт остаётся одним проверяемым профилем:
`susu-hsem-ceit-coursework-v1`.

## Цель

Если текущий профиль окажется полезным, AutoNormoKontrol должен эволюционировать
в движок проверяемых академических документов, способный подключать разные:

- типы работ: курсовая, реферат, отчёт по практике, ВКР;
- кафедры и подразделения;
- нормативные документы;
- локальные решения преподавателей;
- PDF- и, при подтверждённой необходимости, DOCX-backends.

Универсальность не должна ослабить fail-closed модель, трассируемость правил или
воспроизводимость текущего PDF.

## Что является общим, а что профильным

| Общий движок | Профиль документа |
|---|---|
| CLI и поиск зависимостей | Тип работы и обязательная структура |
| Запуск Pandoc, LaTeX и postflight | Формы титульного листа и задания |
| Безопасная semantic Markdown-нотация | Применимый набор нормативных пунктов |
| Механика Draft/Strict | Semantic/external inventory |
| Проверка exact-set и coverage | Параметры оформления и нумерации |
| Трассировка `правило → код → тест` | Библиографическая схема |
| Test harness и mutation contracts | Локальные решения кафедры |
| Хэширование проверяемого снимка | Системная инструкция по содержанию |

Некоторые возможности, например таблицы, формулы и приложения, реализуются в
общем renderer, но профиль определяет, разрешены ли они и какие правила к ним
применяются.

## Архитектурные ограничения уже сейчас

Пока универсализация отложена, новые изменения обязаны соблюдать следующие
границы:

1. Не добавлять кафедры и типы документов цепочками `if/elseif` в общий CLI или
   build-script.
2. Не смешивать пользовательское содержание с фильтрами, стилями и нормативным
   реестром.
3. Не превращать `requirements.json` в общий список, где применимость разных
   профилей определяется неявными договорённостями.
4. Не ослаблять exact-set, coverage и Strict ради повторного использования.
5. Не переносить правило текущей кафедры в «универсальное» только потому, что
   второго профиля пока нет.
6. Сохранять стабильный `profile-id` во всех артефактах приёмки.
7. Любое новое нормативное поведение по-прежнему требует реального обработчика,
   маркера, теста и записи в реестре применимого профиля.
8. Формат профиля не проектировать «на все случаи» без второго реального
   потребителя. Сначала фиксировать seam, затем извлекать проверенную абстракцию.

## Известные точки сцепления текущего профиля

Эти места допустимы в MVP, но должны быть устранены до подключения второго
профиля:

- список `content/*.md` зашит в `scripts/build.ps1`;
- имена `build/coursework.*` используются build-script, CLI и postflight;
- шаблон `templates/susu-coursework.tex` выбирается напрямую;
- exact-set semantic/external ID зашит в `filters/sto-validate.lua`;
- тесты напрямую ссылаются на `susu-coursework.tex` и
  `susu-coursework.sty`;
- `requirements.json` одновременно является нормативным источником и
  эффективным реестром одного профиля;
- стартовые `semantic-review.yaml` и `external-acceptance.yaml` поддерживаются
  вручную;
- обязательная структура metadata и фронтальной части определяется текущим
  шаблоном.

При обычной разработке не требуется немедленно исправлять все эти пункты. Если
изменение касается одного из них, агент должен по возможности локализовать новое
сцепление и не размножать его в дополнительных файлах.

## Целевая модель

```text
engine/
  CLI, build orchestration, generic gates, render primitives, test harness

profiles/
  susu-common/
  susu-hsem-ceit-coursework/
  susu-hsem-ceit-practice-report/
  susu-journalism-coursework/
  susu-law-coursework/
  susu-vkr/

workspaces/
  конкретная работа: content, metadata, bibliography, evidence, assets
```

Эффективный профиль в перспективе должен разрешаться слоями:

```text
общий движок
→ стандарт университета
→ подразделение/кафедра
→ тип документа
→ дисциплина
→ подтверждённый override преподавателя
```

Результат разрешения обязан быть детерминированным и сохраняться в build-отчёте
вместе с версиями слоёв и происхождением каждого override.

## Этапы

### R0 — стабилизировать однопрофильный MVP

Статус: **в работе; текущий приоритет**.

Цель — доказать полезность одного профиля до создания универсальной платформы.

Работы:

- завершить реальную курсовую через цикл Draft → audit → Strict;
- накопить фактические замечания преподавателя и нормоконтроля;
- поддерживать FAQ и agent contract;
- отделить демонстрационное содержание от стартового каркаса;
- добавить безопасное создание новой работы без ручного удаления файлов;
- зафиксировать regression-артефакты текущего PDF и CLI.

Критерий завершения:

- новый пользователь создаёт работу, собирает Draft и понимает блокеры без
  вмешательства разработчика;
- хотя бы одна настоящая работа проходит весь предусмотренный процесс;
- тесты и документация соответствуют фактическому поведению.

### R1 — жизненный цикл отдельной работы

Статус: **следующий практический этап; не требует мультипрофильности**.

Цель — разделить движок и данные конкретного документа до появления второго
профиля.

Работы:

- определить изменяемый слой workspace;
- реализовать `AutoNormoKontrol.cmd new` с защитой от перезаписи;
- создать минимальный Draft-valid starter;
- сбрасывать semantic review в `uncertain`, external acceptance в `pending`;
- добавить smoke-тест «создать workspace → собрать Draft»;
- описать обновление движка без уничтожения пользовательского содержания.

Критерий завершения:

- демонстрационная и новая пользовательская работа не смешиваются;
- команда не удаляет данные без явного подтверждения;
- повторный `new` fail-closed отказывается перезаписывать непустую работу.

### R2 — формальный контракт профиля

Статус: **отложено до появления второго подтверждённого сценария**.

Триггер: принято решение реализовать второй тип документа или кафедру, а не
только гипотетическая возможность.

Работы:

- определить версионируемый `profile.yaml` и его JSON Schema;
- вынести в него пути входов/выходов, тип документа, шаблон и capabilities;
- хранить canonical requirements и review inventories внутри профиля;
- заменить hard-coded списки ID загрузкой эффективного профиля;
- валидировать неизвестные поля и несовместимые версии fail-closed;
- записывать ID и digest эффективного профиля в отчёты сборки.

Критерий завершения:

- текущий профиль собирается через manifest без изменения PDF и диагностики;
- удаление обязательного ID всё так же обнаруживается mutation-тестом;
- не существует fallback на «первый попавшийся» профиль.

### R3 — извлечь текущий профиль без изменения поведения

Статус: **после R2**.

Работы:

- перенести профильные requirements, review templates, prompt, LaTeX template и
  style в единый каталог;
- оставить общие renderer primitives в движке;
- сделать snapshot/regression comparison старой и новой сборки;
- документировать миграцию существующего workspace.

Критерий завершения:

- PDF, предупреждения, ошибки, exit codes и 172-пунктная трассировка текущего
  профиля эквивалентны зафиксированному baseline;
- корневые скрипты не знают имя кафедры или тип `coursework`.

### R4 — второй минимальный профиль как проверка абстракции

Статус: **после R3**.

Рекомендуемый пилот — реальный реферат или близкий отчёт с доступной официальной
методичкой. Не следует начинать с ВКР: она слишком сложна и плохо выявляет,
какая часть сложности относится к архитектуре, а какая — к нормативам.

Работы:

- провести полный нормативный аудит источника;
- создать отдельные requirements, prompt, review inventories и fixtures;
- реализовать `list-profiles`, `new --profile` и сборку выбранного профиля;
- проверить отсутствие утечки требований между двумя профилями.

Критерий завершения:

- оба профиля проходят независимые coverage/mutation suites;
- правило одного профиля не появляется в другом без явного inheritance;
- добавление второго профиля не потребовало ветвления по его ID в engine.

### R5 — кафедральные слои и overrides

Статус: **после успешного второго профиля**.

Работы:

- определить детерминированный порядок наследования;
- регистрировать источник каждого эффективного значения;
- выявлять конфликты одинакового приоритета;
- отделить утверждённые overrides от неподтверждённых пожеланий;
- добавить команду объяснения: `explain <setting-or-rule>`.

Критерий завершения:

- система может ответить, почему применено конкретное поле, интервал или правило;
- конфликт не разрешается молча порядком файлов;
- override преподавателя не меняет базовый профиль кафедры.

### R6 — сложные типы документов

Статус: **долгосрочно**.

Кандидаты реализуются по реальному спросу:

- отчёт по практике: задание, дневник, характеристика и внешние подписи;
- ВКР: формы Б/Г, расширенный normocontrol, иллюстрационные материалы;
- юриспруденция: нормативные акты, судебная практика, сноски;
- журналистика: специальные источники, анализ материалов и приложения.

Каждый такой профиль является отдельным нормативным проектом. Наличие общего
движка сокращает программную работу, но не позволяет копировать требования без
исследования источников.

### R7 — распространение и совместимость

Статус: **после устойчивой мультипрофильности**.

Возможные направления:

- каталог и version constraints профилей;
- upgrade/migration workspace с dry-run;
- GitHub Template/Release для пользователей без агента;
- profile conformance kit для сторонних авторов;
- необязательный DOCX backend с отдельной честной областью гарантий;
- CI как дополнительная воспроизводимая проверка, а не замена локального TeX.

## Протокол задачи для агента

Архитектурная задача из этой roadmap должна начинаться с указания ID этапа,
например `R1/new-workspace`. Агент обязан:

1. прочитать `AGENTS.md`, эту roadmap и обязательный контракт проекта;
2. назвать существующую точку сцепления, которую изменяет задача;
3. зафиксировать неизменяемое текущее поведение;
4. реализовать минимальный вертикальный срез без гипотетических расширений;
5. добавить positive, negative и при необходимости mutation-тест;
6. описать миграцию и обратную совместимость;
7. выполнить полный `AutoNormoKontrol.cmd check`;
8. обновить статус roadmap только при фактическом выполнении критерия этапа.

Запрещено объявлять этап завершённым по наличию manifest или новой папки: должен
работать пользовательский сценарий и сохраняться fail-closed контракт.

## Definition of Done нового профиля

Профиль считается реализованным, только если:

- существует достоверный manifest источников и их хэши;
- каждый нормативный пункт зарегистрирован и классифицирован;
- programmatic правила имеют обработчики и тесты;
- semantic правила имеют prompt и точный review inventory;
- external правила имеют явный acceptance inventory;
- стартовый workspace проходит Draft;
- незакрытый Strict завершается ненулевым exit code;
- полностью закрытый контрольный fixture проходит Strict;
- PDF-postflight и визуальные fixtures покрывают профильные формы;
- документация объясняет пользователю исходные данные и пределы гарантий;
- правила другого профиля не применяются неявно.

## Отложенные решения

Следующие вопросы намеренно не решаются в R0:

- формат inheritance и merge profiles;
- единый каталог профилей или отдельные репозитории;
- поддержка нескольких университетов;
- стабильный API сторонних Lua-фильтров;
- способ обновления engine внутри пользовательского workspace;
- точная модель DOCX backend;
- цифровая подпись semantic/external evidence.

Решение принимается при появлении реального второго потребителя и оформляется
отдельной архитектурной записью, а не скрытым изменением build-script.

## Ближайший фокус

До отдельного решения о начале R2 приоритеты ограничены R0 и R1:

1. доказать текущий end-to-end workflow на настоящей работе;
2. сделать безопасный старт новой курсовой;
3. собирать реальные кафедральные замечания как данные для будущего профиля;
4. не увеличивать перечисленные hard-coded seams.

Это позволяет не оплачивать универсальную архитектуру заранее, но сохраняет
реальный путь к ней без переписывания нормативной модели с нуля.
