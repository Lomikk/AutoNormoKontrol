# Roadmap AutoNormoKontrol

Статус документа: живая архитектурная дорожная карта. Она фиксирует допустимое
направление развития, но не объявляет универсализацию текущим приоритетом.
Текущий продукт остаётся одним проверяемым профилем:
`susu-hsem-ceit-coursework-v1`.

Практические пользовательские решения, из которых выведены границы этапов,
собраны в [`docs/PRODUCT_DECISIONS.md`](PRODUCT_DECISIONS.md). Roadmap задаёт
очерёдность реализации, но не должен переопределять эти продуктовые цели
неявным техническим решением.

## Цель

Если текущий профиль окажется полезным, AutoNormoKontrol должен эволюционировать
в движок проверяемых академических документов, способный подключать разные:

- типы работ: курсовая, реферат, отчёт по практике, ВКР;
- университеты и другие организации со своими нормативными корпусами;
- кафедры и подразделения;
- нормативные документы;
- локальные решения преподавателей;
- PDF- и, при подтверждённой необходимости, DOCX-backends.

Универсальность не должна ослабить fail-closed модель, трассируемость правил или
воспроизводимость текущего PDF.

## Что является общим, а что профильным

| Общий движок | Профиль документа |
|---|---|
| CLI и поиск зависимостей | Тип работы и обязательная структура |
| Запуск Pandoc, LaTeX и postflight | Формы титульного листа и задания |
| Безопасная semantic Markdown-нотация | Применимый набор нормативных пунктов |
| Механика Draft/Strict | Semantic/external inventory |
| Проверка exact-set и coverage | Параметры оформления и нумерации |
| Трассировка `правило → код → тест` | Библиографическая схема |
| Test harness и mutation contracts | Локальные решения кафедры |
| Хэширование проверяемого снимка | Системная инструкция по содержанию |

Некоторые возможности, например таблицы, формулы и приложения, реализуются в
общем renderer, но профиль определяет, разрешены ли они и какие правила к ним
применяются.

Совпадение значения в двух профилях не делает его общим нормативным правилом.
Одинаковые поля или шрифт из разных стандартов могут использовать один
implementation handler, сохраняя две независимые записи требований и два
provenance. Общий нормативный компонент выделяется только при доказанной общей
области действия источника. Полное решение и критерии извлечения зафиксированы в
[`ADR-0001`](decisions/ADR-0001-PROFILE-COMPOSITION.md).

## Архитектурные ограничения уже сейчас

Пока универсализация отложена, новые изменения обязаны соблюдать следующие
границы:

1. Не добавлять кафедры и типы документов цепочками `if/elseif` в общий CLI или
   build-script.
2. Не смешивать пользовательское содержание с фильтрами, стилями и нормативным
   реестром.
3. Не превращать `requirements.json` в общий список, где применимость разных
   профилей определяется неявными договорённостями.
4. Не ослаблять exact-set, coverage и Strict ради повторного использования.
5. Не переносить правило текущей кафедры в «универсальное» только потому, что
   второго профиля пока нет.
6. Сохранять стабильный `profile-id` во всех артефактах приёмки.
7. Любое новое нормативное поведение по-прежнему требует реального обработчика,
   маркера, теста и записи в реестре применимого профиля.
8. Формат профиля не проектировать «на все случаи» без второго реального
   потребителя. Сначала фиксировать seam, затем извлекать проверенную абстракцию.

## Оставшиеся точки сцепления после R3

R2/R3 вынесли список содержания, выходы, шаблон, стиль, фильтры, postflight,
нормативный реестр и canonical ID в manifest и пакет профиля. Общие build,
coverage и traceability больше не знают ID кафедры или тип документа.

До подключения второго профиля остаются осознанные ограничения:

- `profiles/active-profile.txt` выбирает ровно один профиль; публичных команд
  `list-profiles` и `new --profile` ещё нет;
- корневые `semantic-review.yaml` и `external-acceptance.yaml` создаются и
  поддерживаются как evidence текущего workspace; автоматическая инициализация
  из профильных templates относится к R4;
- обязательная структура metadata и фронтальной части всё ещё является
  контрактом конкретного профиля, хотя реализована внутри его пакета;
- часть fixture-набора проверяет именно текущий нормативный профиль и не должна
  ошибочно считаться общей для будущих профилей;
- inheritance, merge и преподавательские override намеренно не определены без
  второго реального потребителя.

Новое сцепление нельзя добавлять ветвлением по `profile_id` в общих скриптах.
Различия следующего типа документа должны сначала выражаться его manifest и
пакетом; только доказанно общая часть переносится в engine.

## Целевая модель до появления второго профиля

```text
engine/
  CLI, build orchestration, generic gates, render primitives, test harness

profiles/
  независимые полные пакеты требований, форм, prompts, filters и tests

workspaces/
  конкретная работа: content, metadata, bibliography, evidence, assets
```

Полный независимый профиль является единицей подключения. До R5 он не наследует
другой профиль и может содержать повторяющиеся декларативные значения, если они
пришли из другого нормативного источника. Исполняемая сборочная логика при этом
не копируется: устойчивые handlers и gates находятся в engine.

Возможная будущая модель после проверки вторым профилем:

```text
полный базовый профиль
+ явный список versioned components
+ подтверждённые local overrides
→ effective profile
```

Это композиция, а не заранее заданное глубокое дерево. Её resolver пока не
реализуется. Если R5 подтвердит необходимость, результат обязан быть
детерминированным и сохраняться в build-отчёте вместе с версиями компонентов,
порядком применения и происхождением каждого override. Конфликт нельзя молча
разрешать порядком файлов.

## Граница publication backend

Текущий проверенный backend остаётся Pandoc → LuaLaTeX/latexmk. Quarto не входит
в MVP и не является решением задачи нормативных профилей. После стабилизации
R1/R4 допустим отдельный сравнительный эксперимент, если он реально удалит
существенную сборочную логику без ухудшения PDF и gates. Requirements,
provenance, semantic/external review и Draft/Strict не должны зависеть от того,
вызывает движок Pandoc напрямую или через иной publication backend.

## Этапы

### Текущий порядок выполнения

Номера R0–R7 являются стабильными идентификаторами областей roadmap, а не
указанием, что все подпункты должны выполняться строго по возрастанию номера.
R2 и R3 уже завершены как фундамент однопрофильного движка. Актуальный
критический путь на текущем этапе:

1. закрыть остаточные вторые источники истины и описать границу workspace (R0);
2. дать автору безопасный ручной context plan для Aider, web-чата и локального
   агента (R1.4a);
3. реализовать создание Draft-valid workspace и стабильную структуру глав
   (R1/workspace);
4. добавить структурированную диагностику с координатой исходника (R1.4b);
5. реализовать явные export/archive без смешивания с disposable `build/`
   (R1/publish);
6. доказать полный coursework workflow на контрольном fixture и реальной работе;
7. только после этого проверять абстракцию вторым небольшим профилем (R4).

R1.2 и R1.3 являются возможностями по реальному спросу и не блокируют этот
путь. R1.4c также не входит в MVP: автоматическое управление Aider допустимо
только после подтверждения полезности ручного context plan и диагностики.

Завершённый фундамент: R2 — формальный контракт профиля; R3 — физическое
извлечение текущего профиля; R1.1 — минимальный воспроизводимый asset pipeline.
Подробности завершённых миграций сохраняются ниже как история архитектурных
решений, но не считаются текущей очередью работ.

### R0 — стабилизировать однопрофильный MVP

Статус: **в работе; текущий приоритет**.

Цель — доказать полезность одного профиля до создания универсальной платформы.

Работы:

- завершить реальную курсовую через цикл Draft → audit → Strict;
- накопить фактические замечания преподавателя и нормоконтроля;
- поддерживать FAQ и agent contract;
- стабилизировать структуру однопрофильного репозитория перед новыми функциями:
  хранить нормативный корпус только в каноническом `sources/`, не версионировать
  входную папку импорта и удалять осиротевшие каталоги после миграций;
- миграция происхождения СТО ЮУрГУ 21–2008 выполнена 15 июля 2026 года:
  временный корневой `Normokontrol-2008-2017.pdf` заменён каноническим
  оригиналом из `sources/susu/sto-21-2008/`. Оба PDF имеют 56 страниц, а
  нормализованные тексты и действовавший TXT совпали точно; `format-spec.yaml`,
  нормативный реестр и документация переведены на единый manifest корпуса;
- второй источник истины списка content-файлов устранён 15 июля 2026 года:
  document snapshot требует явный непустой `ContentPaths`, штатная сборка
  передаёт его из manifest профиля, а отсутствие fallback закреплено
  regression-тестами;
- назначение верхнеуровневых каталогов и граница между engine, profile,
  workspace, sources, tests и generated `build/` зафиксированы в
  `docs/PROJECT_STRUCTURE.md`, включая безопасную матрицу контекста Aider;
- отделить демонстрационное содержание от стартового каркаса;
- добавить безопасное создание новой работы без ручного удаления файлов;
- зафиксировать regression-артефакты текущего PDF и CLI.

Критерий завершения:

- новый пользователь создаёт работу, собирает Draft и понимает блокеры без
  вмешательства разработчика;
- в репозитории нет двух версионируемых копий одного нормативного корпуса,
  осиротевших путей предыдущей архитектуры и скрытого списка входов в обход
  manifest активного профиля;
- миграция нормативного первоисточника воспроизводимо объясняет, какой PDF и
  текстовый корпус доказывают каждый пункт текущего профиля; удаление старого
  оригинала не оставляет битых ссылок или непроверенного изменения требований;
- хотя бы одна настоящая работа проходит весь предусмотренный процесс;
- тесты и документация соответствуют фактическому поведению.

Наблюдения из реального Draft-цикла, ожидающие отдельного решения:

- `STO-8.2.8` и `STO-8.2.9` действительно запрещают отрыв заголовка от текста и
  требуют после заголовка внизу страницы не менее трёх строк. Текущий профиль
  использует `Needspace`, но перед выпуском нужен PDF regression-test не только
  реализации, но и фактического расположения каждого заголовка;
- устное требование начинать каждый заголовок первого уровня, включая
  заключение и библиографический список, с новой страницы не найдено в тексте
  СТО ЮУрГУ 21–2008. В СТО новая страница явно предписана для каждого
  приложения (`STO-7.12.3`). Для основных разделов это требование нельзя
  объявлять нормативным без подтверждения кафедры или преподавателя;
- устное ограничение «перед новым разделом нельзя оставлять более половины
  страницы пустой» также не найдено в СТО и пока регистрируется как кандидат на
  локальный override. После подтверждения геометрическую часть можно проверять
  в PDF postflight, но автоматическое дописывание или удаление текста запрещено:
  содержательное сокращение/расширение выполняется автором или ИИ и проходит
  отдельный смысловой аудит;
- при формализации правила нужно определить измеримую метрику: считать строки,
  заполненную высоту текстовой области либо долю пустой площади; отдельно
  описать страницы с рисунками, таблицами, формулами и намеренными разрывами.

### R1 — жизненный цикл отдельной работы

Статус: **следующий практический этап; не требует мультипрофильности**.

Цель — разделить движок и данные конкретного документа до появления второго
профиля.

Этап состоит из двух последовательных срезов:

- `R1/workspace`: граница изменяемых данных, Draft-valid starter, команда `new`
  и стабильные имена файлов глав;
- `R1/publish`: команды `export` и `archive`, которые работают только с
  результатом уже состоявшейся сборки и не меняют содержание или evidence.

Отсутствие этих команд является известным ограничением текущего CLI: новую
работу пока приходится подготавливать, а готовый PDF копировать из `build/`
вручную. До реализации команд это ограничение должно быть явно указано в
пользовательской документации, а не скрыто примером несуществующей команды.

Работы:

- определить изменяемый слой workspace;
- реализовать `AutoNormoKontrol.cmd new` с защитой от перезаписи;
- создать минимальный Draft-valid starter;
- разделить содержание starter и реальной работы по устойчивым смысловым
  единицам: один Markdown-файл на главу либо крупный раздел, но не на страницу
  и не на каждый мелкий подраздел;
- использовать стабильные имена файлов, не зависящие от редактируемого
  заголовка главы. Рекомендуемая схема: `00-introduction.md`,
  `10-chapter-01.md`, `20-chapter-02.md`, `90-conclusion.md`,
  `99-appendices.md`; заголовок документа хранится только внутри Markdown, а
  порядок входов задаётся manifest активного профиля и не выводится из имени;
- разделить технический и пользовательский выходы: `draft` продолжает собирать
  один воспроизводимый PDF в disposable `build/`, а явная команда `export`
  публикует последний успешно проверенный PDF в понятный `output/` с именем,
  определённым профилем и metadata;
- не создавать PDF с временной меткой при каждой Draft-сборке и не хранить
  автоматическое кольцо «последних пяти PDF». История редактируемых исходников
  остаётся задачей Git; PDF-архив создаётся только по явной команде `archive`
  для версии, реально отправленной преподавателю или предназначенной для печати;
- export не должен мешать дальнейшему Draft, если пользователь открыл ранее
  экспортированный PDF в Adobe или другом просмотрщике: Draft работает только с
  `build/`, а блокировка файла `output/` даёт понятный отказ именно команды
  export без потери собранного результата;
- сбрасывать semantic review в `uncertain`, external acceptance в `pending`;
- добавить smoke-тест «создать workspace → собрать Draft»;
- описать обновление движка без уничтожения пользовательского содержания.

Критерий завершения:

- демонстрационная и новая пользовательская работа не смешиваются;
- переименование главы внутри Markdown не требует переименования файла или
  изменения ссылок сборки, а агент может добавить в контекст только затронутую
  главу и обязательные read-only контракты;
- после успешного Draft пользователь может экспортировать один понятный PDF из
  `output/`, не просматривая служебные файлы `build/`; повторный export имеет
  явную политику перезаписи, а archive создаётся только по явному действию;
- команда не удаляет данные без явного подтверждения;
- повторный `new` fail-closed отказывается перезаписывать непустую работу.

### R1.1 — воспроизводимые ассеты документа

Статус: **минимальный вертикальный срез выполнен 15 июля 2026 года; дальнейшие
типы ассетов добавляются только по реальному спросу**.

Цель — сделать изображения, схемы, диаграммы и графики частью проекта, а не
непрозрачными файлами, разбросанными по компьютеру или созданными вручную в
Word. Markdown продолжает описывать только смысловую ссылку на готовую
иллюстрацию; он не получает raw TeX, Python или команды сборки.

Минимальный вертикальный срез:

- ввести документированный каталог `assets/` с разделением как минимум на
  `images/` (статические изображения), `diagrams/` (исходники диаграмм),
  `data/` (входные наборы данных) и `source/` (редактируемые исходники вроде
  draw.io/SVG, если они существуют);
- считать `build/assets/` только производным выходом: его можно удалить и
  воспроизвести, а содержимое туда не редактируется вручную;
- описать явный manifest ассета: ID, тип, исходные файлы, генератор, ожидаемый
  выход, источник/лицензия и, для графика, набор данных;
- добавить один безопасный asset-builder с белым списком поддерживаемых
  генераторов. На первом шаге достаточно статического PNG/JPEG/PDF и одного
  `.tex`-исходника диаграммы, собранного в PDF;
- включать в отчёт сборки хэши manifest, входов и готового ассета, а при
  подключении ассетов расширить проверяемый снимок документа: изменение
  рисунка или его данных не должно оставлять актуальным старый semantic review;
- проверять отсутствие файлов, неизвестных ID, циклов зависимостей и
  неактуальных generated outputs; для каждого правила добавить обычный
  положительный и отрицательный fixture;
- оставить программным проверкам структуру: существование файла, ID,
  упоминание до рисунка, тип, разрешённые атрибуты и источник данных. Оси,
  масштаб, читаемость, смысл подписи и соответствие данным остаются частью
  semantic/external review.

Не делать на этом этапе: произвольный запуск файлов из `assets/`, сетевую
загрузку изображений при сборке, хранение единственного исходника в
`build/`, автоматическую «угадай-график» генерацию по тексту и поддержку всех
графических форматов сразу.

Критерий завершения:

- один реальный рисунок или диаграмма воспроизводится на чистой копии проекта
  из versioned source и попадает в PDF;
- изменение его входа делает проверяемый снимок устаревшим;
- удаление исходника, manifest-записи или generated PDF даёт понятный
  fail-closed отказ, а не старую картинку из кэша.

Результат: реализованы versioned CSV и TeX/pgfplots-источник, белый список
генераторов, manifest, воспроизводимый `build/assets/*.pdf`, asset report и
включение исходников/выхода в `content-hash`. Positive, negative, mutation и
byte-reproducibility тесты входят в штатный `check`; прямая сборка
`fixtures/architecture.tex` из обычного build удалена.

### R1.2 — контролируемые листинги кода

Статус: **отложено до реального требования преподавателя или курсовой с
фрагментами исходного кода**.

Цель — не пропускать Markdown fenced code blocks как случайный побочный эффект
Pandoc, а ввести отдельный семантический объект «листинг» с явными правилами
и границами гарантий.

Работы:

- сначала зафиксировать решение кафедры: является ли листинг допустимым,
  нумеруется ли он как рисунок/самостоятельный объект, нужны ли ссылки и
  подписи, какой допустим шрифт, размер и цвет;
- до появления этого контракта запретить неописанные code blocks fail-closed;
- ввести ограниченную нотацию листинга с ID, языком, подписью и источником
  фрагмента; raw TeX, HTML и пользовательские параметры LaTeX не разрешать;
- реализовать renderer и стиль отдельно от content, начать с
  чёрно-белого моноширинного представления без `shell-escape` и без внешнего
  Pygments/minted;
- добавить тесты на нумерацию/ссылку/подпись/переполнение и на запрет
  неразрешённых атрибутов; semantic review проверяет, что фрагмент нужен,
  соответствует программе и не содержит секретов.

Подсветка синтаксиса, цветной фон, Consolas и уменьшенный кегль — не свойства
«по умолчанию». Это будущая настройка профиля только после подтверждения
допустимости; базовая версия не должна незаметно нарушить основной стиль
документа.

### R1.3 — данные как исходный код и расчётные параметры

Статус: **отложено до первой работы, где одно значение реально влияет на
несколько таблиц, формул, графиков или выводов**.

Цель — заменить ручное копирование результатов из Excel/диалогов с ИИ
воспроизводимой цепочкой «входные данные → расчёт → таблица/график/текст».
Это не означает, что каждое число в прозе становится переменной.

Работы:

- хранить versioned исходные данные в workspace: исходный файл как снимок
  (`data/raw/`), канонический машиночитаемый ввод (`data/input/`) и результаты
  расчёта (`data/generated/`); Excel остаётся удобным редактором или
  первичным снимком, но не скрытым вычислительным движком сборки;
- добавить детерминированные генераторы расчётов в `scripts/` или
  `assets/plots/`, с фиксированными входами, версиями и понятным запуском;
- создать небольшой реестр параметров только для величин с несколькими
  производными употреблениями: стабильный ID, значение, единица измерения,
  точность, источник, дата/версия и область применимости;
- связывать график, таблицу и вычисленный результат через manifest и
  provenance-отчёт, а не через неявные формулы Excel или просьбу ИИ «поменять
  число везде»;
- добавить data fingerprint в build/report и зависимость его от snapshot для
  semantic review; изменение входного CSV, параметра, расчётного скрипта или
  generated figure обязано быть видно как изменение проверяемой версии;
- тестировать неизвестный параметр, отсутствующий вход, изменённые без
  пересчёта результаты, единицы измерения и расхождение manifest с выходом.

Граница против оверинжиниринга: единичный факт или число, встречающееся на
нескольких страницах, обычно меняется контролируемым поиском и обзором ИИ.
Реестр нужен, когда есть производные вычисления либо риск расхождения таблиц,
графиков и выводов. Автоматическая подстановка `{{параметр}}` в свободный текст
не вводится без отдельного контракта типов, единиц, форматирования и тестов.

Критерий завершения:

- один реальный набор данных порождает минимум один результат и один график
  повторяемо на чистой копии;
- изменение входа не позволяет выдать старые таблицы/графики за актуальные;
- provenance отчёт объясняет происхождение каждого опубликованного результата
  без обращения к чату ИИ или локальной истории Excel.

### R1.4 — структурированная диагностика для ИИ

Статус: **`R1.4a/context-plan-v1` реализован 15 июля 2026 года; диагностика
`R1.4b` остаётся следующей задачей этого блока и не требует мультипрофильности**.

Цель — вместо передачи ИИ многотысячного Pandoc/TeX-лога выдавать компактные
стабильные коды и машиночитаемый контекст, достаточный для безопасной локальной
правки. Пользователь должен иметь возможность передать агенту одну строку ошибки
либо `build/diagnostics.json`, не объясняя внутреннее устройство сборки.

R1.4 выполняется тремя отдельными задачами, а не одним большим изменением:

- `R1.4a/context-plan-v1`: provider-neutral список editable, read-only и
  excluded файлов плюс готовое копируемое представление `/drop`, `/read-only`,
  `/add`; никакого управления процессом Aider;
- `R1.4b/diagnostics-v1`: стабильные коды, JSON Schema, координаты
  `file:line`, `object_id`, фаза и безопасная context suggestion;
- `R1.4c/aider-adapter`: автоматическое применение перехода контекста и
  ограниченный repair-loop только после проверки первых двух ручных срезов на
  настоящей работе.

Сначала определяется граница workspace и реализуется R1.4a. Диагностика R1.4b
использует уже готовую политику контекста и не изобретает собственный список
разрешённых файлов. R1.4c не должен блокировать завершение coursework MVP.

Реализованный срез `R1.4a/context-plan-v1`:

- фиксирует пять прав доступа вместо растущей task taxonomy: `edit-content`,
  `edit-references`, `edit-metadata`, `design-structure`, `review-content`;
- строит пути только из активного `profile.yaml`, а target принимает только из
  его `inputs.content`;
- выпускает детерминированные `build/ai/context-plan.json`, планы переходов и
  отдельное Aider-представление `/drop`, `/read-only`, `/add`;
- использует пользовательский `/load` как authority boundary и не управляет
  процессом Aider;
- загружает для каждого switch отдельную capability-инструкцию, чтобы после
  перехода модель не видела устаревшее название текущего режима;
- отклоняет неизвестный capability, неизвестный или физически существующий вне
  profile target, traversal и попытку editable-доступа к protected-областям;
- покрыт отдельными contract-тестами и общей командой `check`.

Срез не классифицирует естественный язык, не анализирует Draft diagnostics, не
запускает Aider и не реализует repair-loop. Эти границы остаются за R1.4b/R1.4c.

Минимальный вертикальный срез строится на уже наблюдавшихся отказах:

- отсутствующий ключ цитирования;
- длинный URL, `Overfull \hbox` и пересечение поля PDF;
- таблица без ID, подписи или первого упоминания;
- приложение без ссылки в основном тексте;
- ошибка этапа asset-builder, Pandoc/Lua filter, LaTeX или PDF-postflight.

Работы:

- определить версионируемую JSON Schema диагностики и единый реестр стабильных
  кодов проекта, не зависящих от языка сообщения и текста стороннего инструмента;
- записывать `build/diagnostics.json` при успешной и неуспешной сборке. Для
  каждого события хранить минимум `code`, `severity`, `phase`, `rule_ids`,
  `message`, `file`, диапазон строк/колонок, `object_id`, краткое evidence и
  разрешённую область исправления;
- считать стабильный `object_id` основной координатой объекта, путь и диапазон
  строк — координатой текущей версии исходника, а страницу PDF — только
  дополнительной координатой для визуальной проверки. После правки строки могут
  сдвигаться, поэтому одна страница или один номер строки не являются устойчивым
  идентификатором;
- для ошибок PDF-postflight восстанавливать исходник через source map, стабильный
  ID либо точный короткий фрагмент текста. При неоднозначном совпадении выводить
  кандидатов и не приписывать ошибку произвольному Markdown-файлу;
- связывать внутренний код AutoNormoKontrol с нормативными ID отдельно: например,
  один `ANK-TABLE-MISSING-ID` может ссылаться на `STO-object-id` и
  `STO-8.6.3`, но номер СТО не заменяет тип технической ошибки;
- нормализовать диагностику стадий `profile`, `assets`, `markdown`, `pandoc`,
  `latex`, `postflight` и `compliance`, сохраняя исходный лог отдельным
  артефактом для разработчика;
- выводить в консоль короткую копируемую строку, например:

  ```text
  ANK-TABLE-MISSING-ID [STO-8.6.3] content/99-appendix.md:7
  ```

- добавить для каждого кода контролируемую справку: причина, безопасное
  исправление, запрещённые обходы и минимальный набор editable/read-only файлов
  для Aider. Справка не должна содержать исполняемые shell-команды из входного
  документа;
- считать Aider пользовательским редактором содержания, а не средством
  разработки AutoNormoKontrol. Обычные сценарии Aider могут редактировать только
  разрешённые файлы workspace (`content/*.md`, а по типу задачи также
  `bibliography.bib` или `metadata.yaml`); `sources/**`, `profiles/**`,
  `scripts/**`, `tests/**`, `fixtures/**`, нормативные реестры и журналы
  приёмки не предлагаются ему для изменения;
- ввести декларативную политику контекста для задач написания, библиографии,
  метаданных и исправления Draft. Она должна различать editable, обязательные
  read-only и не нужные по умолчанию файлы; чтение нормативного первоисточника
  не запрещается технически, но не расходует обычный авторский контекст и
  остаётся задачей разработки профиля вне Aider-workflow;
- формировать из этой политики provider-neutral `context plan`, пригодный не
  только для Aider, но и для локальных агентов с доступом к workspace и для
  браузерного copy-paste workflow. План должен содержать тип задачи, активный
  профиль и его digest, editable/read-only/excluded пути, причины включения и
  срок применимости; конкретные команды клиента являются отдельным
  представлением этого плана;
- показывать рядом с пользовательской диагностикой необязательную подсказку
  контекста, сформированную только из разрешённых путей, например:

  ```text
  AI context suggestion:
    /add content/20-chapter-02.md
  ```

  В JSON эта рекомендация хранится как структурированные списки editable и
  read-only файлов, а команда `/add` является лишь представлением для Aider, не
  частью стабильной модели диагностики;
- первым пользовательским срезом сделать только генерацию готового копируемого
  блока `/drop`, `/read-only` и `/add`. Не управлять процессом Aider, не вводить
  собственный агентский цикл и не требовать от пользователя знания структуры
  репозитория до подтверждения полезности этого простого интерфейса;
- после подтверждения ручного среза разрешить отдельному Aider-адаптеру
  автоматически применять только переход контекста: убрать предыдущую главу,
  сохранить обязательные read-only контракты и добавить следующую разрешённую
  главу. Автоматический переход не означает разрешение самостоятельно выбирать
  тему, писать содержание, принимать diff или изменять статусы приёмки;
- выполнять переход контекста только по явному действию пользователя либо после
  успешного завершения определённого этапа. При незакоммиченных изменениях,
  ошибочном Draft, неизвестном пути или несовпадении profile digest переход
  должен останавливаться fail-closed с копируемой ручной рекомендацией;
- для полноценных локальных агентов отдавать тот же context plan как ограничение
  области работы: агент может читать дополнительные файлы для диагностики, но
  обязан объяснить расширение контекста и не получает права редактировать
  excluded-пути. Это дополняет, а не заменяет `AGENTS.md` и системный prompt;
- предусмотреть режим, в котором агент читает diagnostics JSON, изменяет только
  перечисленные workspace-файлы, повторяет Draft и останавливается при успехе,
  неизменившемся наборе ошибок или лимите итераций;
- не разрешать такому циклу автоматически изменять профиль, нормативный реестр,
  `semantic-review.yaml`, `external-acceptance.yaml` или выставлять внешние
  статусы;
- не включать персональные данные и большие фрагменты содержания в компактный
  отчёт без необходимости; пути должны оставаться относительными workspace;
- сохранить фактический ненулевой exit code сборки: наличие удобного JSON не
  превращает ошибочный PDF в успешный результат.

Тесты:

- один fixture на каждый код минимального среза и точное соответствие
  `code → phase → rule_ids → source location`;
- тесты source mapping для переставленных строк, одинакового текста в двух
  главах и ошибки postflight, для которой известна только страница PDF;
- детерминированный порядок нескольких ошибок и отсутствие дубликатов одного
  первичного дефекта, проявившегося в разных инструментах;
- одинаковый код при русской и английской локали внешней программы;
- неизвестный код и несовместимая версия schema отклоняются fail-closed;
- mutation-тест не позволяет диагностике рекомендовать изменение запрещённого
  файла или обход Strict;
- policy-тесты не позволяют авторскому Aider-сценарию добавлять в editable
  контекст `sources/**`, `profiles/**`, `scripts/**`, журналы приёмки или
  generated-файлы из `build/**`;
- contract-тест подтверждает, что один и тот же context plan детерминированно
  преобразуется в Aider-команды и понятный список файлов для локального агента;
- transition-тест сохраняет обязательные read-only файлы, удаляет только
  предыдущий авторский контекст и отказывается продолжать при ошибочном Draft
  или несовпадении digest профиля;
- smoke-сценарий «сломанный Markdown → diagnostics JSON → разрешённая правка →
  успешный Draft».

Критерий завершения:

- для ошибок минимального среза пользователь копирует не более одной короткой
  строки или передаёт один JSON-файл, а агент однозначно находит место и область
  исправления;
- пользовательский вывод не содержит полного TeX-лога по умолчанию, но исходная
  диагностика остаётся доступной разработчику;
- автоматизированный repair-loop не выходит за разрешённые файлы и не может
  ослабить нормативные gates;
- коды и schema документированы как стабильный контракт с версией и
  regression-тестами.

### R4 — второй минимальный профиль как проверка абстракции

Статус: **после закрытия coursework workflow R0/R1; R3 уже выполнен**.

Триггер этапа — не только наличие извлечённого профиля, но и стабильная граница
workspace, Draft-valid `new`, проверенный ручной context workflow, regression
baseline текущего профиля и контрольный end-to-end Strict fixture. Реальная
курсовая может оставаться заблокированной внешними evidence; это не заменяет
контрольный fixture и не разрешает ослаблять Strict.

Рекомендуемый пилот — реальный реферат или близкий отчёт с доступной официальной
методичкой. Не следует начинать с ВКР: она слишком сложна и плохо выявляет,
какая часть сложности относится к архитектуре, а какая — к нормативам.

Работы:

- провести полный нормативный аудит источника;
- создать независимые requirements, prompt, review inventories и fixtures;
- разрешить взять соседний профиль как стартовую реализацию, но заново проверить
  применимость каждого требования: копия реестра не является нормативным
  аудитом нового типа документа;
- сначала сохранить профиль полным и самодостаточным, не создавая
  предположительный `susu-common` ради удаления одинаковых значений;
- построить дифференциальную матрицу из ADR-0001: источник, область действия,
  mechanism, нормализованное правило, параметры и implementation key;
- извлечь одинаковый исполняемый обработчик в engine, если совпала техническая
  семантика; общий нормативный компонент не выделять без доказанной общей
  области действия источника;
- реализовать `list-profiles`, `new --profile` и сборку выбранного профиля;
- проверить отсутствие утечки требований между двумя профилями.

Если `profile-v1` мешает выразить второй профиль, допустима одна контролируемая
миграция обоих пакетов на `profile-v2`. Бессрочная поддержка старой схемы не
является целью, но до и после миграции сравниваются PDF, diagnostics, exit codes
и traceability baseline текущей курсовой.

Критерий завершения:

- оба профиля проходят независимые coverage/mutation suites;
- правило одного профиля не появляется в другом без явного выбора компонента;
- добавление второго профиля не потребовало ветвления по его ID в engine;
- одинаковый существенный обработчик не существует в двух расходящихся копиях;
- матрица объясняет, почему повторяющиеся requirement records оставлены
  независимыми либо почему реализация была извлечена в engine.

### R5 — явная композиция и overrides

Статус: **только после успешного второго профиля и доказанной потребности в
композиции**.

Работы:

- определить версионируемый формат явного списка компонентов; не выводить
  порядок из пути, имени файла или предположительной иерархии каталогов;
- определить детерминированный порядок композиции без глубокого неявного
  наследования и запретить silent last-write-wins;
- регистрировать источник каждого эффективного значения;
- выявлять конфликты одинакового приоритета;
- отделить утверждённые overrides от неподтверждённых пожеланий;
- позволить одному versioned-компоненту устойчивой области действия содержать
  несколько тематических правил, не создавая микропакет на каждый параметр;
- не фиксировать заранее, сколько компонентов может подключать одна работа. При
  появлении реального случая множественности хранить явный список и
  версии/digests в workspace, не создавая профиль на каждую комбинацию;
- пересчитывать exact-set, effective profile digest и review inventories так,
  чтобы компонент не мог ослабить обязательный gate молча;
- добавить команду объяснения: `explain <setting-or-rule>`.

Критерий завершения:

- система может ответить, почему применено конкретное поле, интервал или правило;
- конфликт не разрешается молча порядком файлов;
- override преподавателя не меняет базовый профиль кафедры;
- удаление либо перестановка компонента меняет effective digest и делает старые
  evidence неактуальными;
- один override-набор может изменять несколько тематических параметров без
  копирования полного профиля; если подтверждена необходимость нескольких
  наборов, несовместимые значения завершаются понятным fail-closed отказом.

### R6 — сложные типы документов

Статус: **долгосрочно**.

Кандидаты реализуются по реальному спросу:

- отчёт по практике: задание, дневник, характеристика и внешние подписи;
- ВКР: формы Б/Г, расширенный normocontrol, иллюстрационные материалы;
- юриспруденция: нормативные акты, судебная практика, сноски;
- журналистика: специальные источники, анализ материалов и приложения.

Каждый такой профиль является отдельным нормативным проектом. Наличие общего
движка сокращает программную работу, но не позволяет копировать требования без
исследования источников.

### R7 — распространение и совместимость

Статус: **разделено на раннюю однопрофильную поставку и позднюю
мультипрофильную совместимость**.

Возможные направления:

- после стабильного R1: GitHub Template/Release, инструкция установки и
  минимальный starter для пользователей без агента;
- после R4/R5: каталог и version constraints профилей;
- после R4/R5: namespaced-профили других университетов с независимыми
  нормативными корпусами и conformance tests;
- после R4/R5: upgrade/migration workspace с dry-run;
- profile conformance kit для сторонних авторов;
- необязательный DOCX backend с отдельной честной областью гарантий;
- CI как дополнительная воспроизводимая проверка, а не замена локального TeX.

## Протокол задачи для агента

Архитектурная задача из этой roadmap должна начинаться с указания ID этапа,
например `R1/new-workspace`. Агент обязан:

1. прочитать `AGENTS.md`, эту roadmap и обязательный контракт проекта;
2. назвать существующую точку сцепления, которую изменяет задача;
3. зафиксировать неизменяемое текущее поведение;
4. реализовать минимальный вертикальный срез без гипотетических расширений;
5. добавить positive, negative и при необходимости mutation-тест;
6. описать миграцию и обратную совместимость;
7. выполнить полный `AutoNormoKontrol.cmd check`;
8. обновить статус roadmap только при фактическом выполнении критерия этапа.

Запрещено объявлять этап завершённым по наличию manifest или новой папки: должен
работать пользовательский сценарий и сохраняться fail-closed контракт.

## Definition of Done нового профиля

Профиль считается реализованным, только если:

- существует достоверный manifest источников и их хэши;
- каждый нормативный пункт зарегистрирован и классифицирован;
- programmatic правила имеют обработчики и тесты;
- semantic правила имеют prompt и точный review inventory;
- external правила имеют явный acceptance inventory;
- стартовый workspace проходит Draft;
- незакрытый Strict завершается ненулевым exit code;
- полностью закрытый контрольный fixture проходит Strict;
- PDF-postflight и визуальные fixtures покрывают профильные формы;
- документация объясняет пользователю исходные данные и пределы гарантий;
- правила другого профиля не применяются неявно.

## Отложенные решения

Следующие вопросы намеренно не решаются в R0:

- точный формат manifest/lock для явной композиции компонентов;
- единый каталог профилей или отдельные репозитории;
- способ публикации, поиска и доверия сторонним профилям других университетов;
- стабильный API сторонних Lua-фильтров;
- способ обновления engine внутри пользовательского workspace;
- точная модель DOCX backend;
- целесообразность Quarto как сменного publication backend;
- цифровая подпись semantic/external evidence.

Решение принимается при появлении реального второго потребителя и оформляется
отдельной архитектурной записью, а не скрытым изменением build-script.

## Ближайший фокус

После завершения R2/R3 и минимального R1.1 практический фокус находится на
R0/R1. Ближайшие задачи выполняются в таком порядке:

1. `R1.4a/context-plan-v1` — реализовано; ручная проверка `/load` в Aider остаётся
   пользовательским acceptance-сценарием;
2. реализовать R1/workspace с безопасными `new` и Draft-valid starter;
3. добавить R1.4b/diagnostics-v1 с `file:line`, `object_id` и context suggestion;
4. реализовать R1/publish (`export`, затем явный `archive`);
5. доказать engine workflow закрытым Strict fixture и пройти реальный цикл
   курсовой настолько, насколько позволяют настоящие внешние evidence;
6. сохранить regression baseline и только затем выбрать второй небольшой тип
   документа для R4.

Документационный аудит выполнен 15 июля 2026 года: список CLI сведён в
`docs/PROJECT_STRUCTURE.md`, отсутствие `new/export/archive` отмечено явно,
добавлен hand-off Draft → content-hash → semantic review → external acceptance
→ Strict и устранено предположение, что пользователь сам угадывает главу для
Aider. 15 июля 2026 года добавлен `context-plan-v1`; следующая незакрытая задача
ближайшего фокуса — R1/workspace с безопасными `new` и Draft-valid starter.

R3 остаётся чистой миграцией. Реальный второй тип документа, resolver композиции
и слои override не были в него добавлены и остаются задачами последующих этапов.

Архитектурное решение от 16 июля 2026 года уточняет терминологию: целевой
механизм R5 — явная композиция полных профилей и versioned components, а не
заранее заданное глубокое inheritance-дерево. Поддержка других университетов
разрешена направлением развития, но её каталог и модель доверия остаются R7.
