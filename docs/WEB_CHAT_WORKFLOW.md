# Работа через браузерный ИИ-чат и Aider

Этот режим нужен, когда ИИ не видит локальный репозиторий, не может изменять
файлы и запускать AutoNormoKontrol. Он менее удобен, чем полноценный агент, но
может оставаться проверяемым, если локальная сборка считается единственным
источником результата.

Канонический список реально существующих CLI-команд и границы каталогов
находятся в [`docs/PROJECT_STRUCTURE.md`](PROJECT_STRUCTURE.md). Жизненный цикл
`new → draft → export → archive` реализован; новую работу создают центральной
командой `AutoNormoKontrol.cmd new <название>`, а остальные команды запускают
через тонкий launcher внутри созданного workspace.

## Разделение ответственности

Браузерный ИИ:

- анализирует переданные файлы и указания;
- предлагает новый текст или точный патч;
- объясняет, какие сведения отсутствуют;
- не утверждает, что локальные тесты прошли, если не видел их фактический вывод;
- не принимает внешние решения от имени пользователя или кафедры.

Пользователь локально:

- применяет изменения;
- запускает `AutoNormoKontrol.cmd draft` или `check`;
- просматривает PDF;
- возвращает чату ошибки, diff и необходимые исходники;
- принимает внешние решения только при наличии реального основания.

## Что передать в браузерный чат

Лучший вариант — приложить архив только каталога конкретной работы без `.git/`,
`build/` и `output/`, если сервис поддерживает архивы и политика
конфиденциальности допускает загрузку. Центральный engine, `profiles/`,
нормативные sources и тесты передавать для обычного написания не нужно. Перед
отправкой нужно удалить персональные данные, которые не должны попадать во
внешний сервис.

Если передача идёт отдельными файлами, минимальный пакет зависит от задачи.

### Для обычной правки содержания

Обязательный контекст:

- `AGENTS.md`;
- `profiles/active-profile.txt`;
- указанный им `profiles/<profile-id>/profile.yaml`;
- файлы из полей `compliance.system_prompt` и `compliance.requirements`
  выбранного manifest;
- `README.md`;
- `metadata.yaml`;
- изменяемый файл `content/*.md`;
- связанные главы, если задача затрагивает цель, задачи, результаты или выводы;
- `bibliography.bib`, если используются или добавляются источники.

Дополнительно передаются изображения, таблицы исходных данных, задание и
замечания преподавателя, если без них нельзя проверить смысл.

### Для диагностики ошибки сборки

Нужны:

- полный текст команды и вывода от первого сообщения об ошибке до exit code;
- `git diff` текущих изменений;
- файлы, названные в ошибке;
- для ошибки валидатора — соответствующий фрагмент Markdown и номер STO;
- для ошибки LaTeX — `build/coursework.log` и при необходимости
  `build/coursework.tex`, но не весь каталог `build/` без причины.

### Для финального semantic review

Нужно передать один неизменяемый снимок:

- все `content/*.md`;
- `metadata.yaml`;
- `bibliography.bib`;
- `compliance/semantic-review.yaml`;
- относящиеся к проверкам изображения;
- `Content hash` из штатной Draft-сборки;
- обязательные контрактные файлы.

Hand-off перед Strict выполняется только в таком порядке: успешный Draft,
фиксация его `content-hash`, полный semantic review этого же снимка, реальное
внешнее решение для external acceptance, затем Strict. Успешный Draft не
является доказательством semantic review, external acceptance или готовности
Strict.

После аудита нельзя применять предложенные изменения и одновременно сохранять
полученный `pass`: любая правка создаёт новый хэш и требует новой проверки.

## Ручной copy-paste без Aider

1. Зафиксировать чистую исходную точку командой `git status` и при необходимости
   сделать локальный commit.
2. Передать чату контракт и только относящиеся к задаче файлы.
3. Попросить вернуть полный файл для короткого документа либо unified diff для
   локальной правки. Не принимать ответы с пропусками вида «остальной текст без
   изменений».
4. Просмотреть патч и применить его локально.
5. Выполнить `AutoNormoKontrol.cmd draft`; при изменении программы — `check`.
6. Передать чату точный вывод ошибки и новый `git diff`, если цикл продолжается.
7. Перед сохранением результата проверить `git diff --check` и сам PDF.

Шаблон первого запроса:

```text
Ты редактируешь проект AutoNormoKontrol без доступа к моему компьютеру.
Обязательные правила находятся в приложенных AGENTS.md, системной инструкции,
requirements.json и README.md. Не выдумывай факты и не меняй semantic-review или
external-acceptance без отдельной задачи.

Задача: [что изменить].
Подтверждённые исходные данные: [факты].
Файлы, которые разрешено менять: [список].
Верни unified diff без сокращений. После применения я локально запущу Draft и
пришлю фактический вывод.
```

## Aider как локальный мост к браузерному чату

Aider удобнее ручного копирования, потому что формирует контекст репозитория и
применяет ответ браузерной модели к локальным файлам. Это не даёт браузерному ИИ
доступ к компьютеру и не отменяет локальные тесты.

Схема `R1.4a/context-plan-v1` ниже сохранена для встроенного legacy workspace и
ручных экспериментов с Aider. В отдельном workspace курсовой команда `context`
намеренно не входит в MVP: Gemini CLI работает от корня одной работы по
локальным `GEMINI.md` и `AGENTS.md`, а управление контекстом OpenCode/Aider
исследуется позднее.

Для legacy workspace:

1. из корня проекта один раз подготовить разрешённый контекст, например:

   ```powershell
   .\AutoNormoKontrol.cmd context edit-content content/00-introduction.md
   ```

2. внутри Aider выполнить ровно показанную CLI команду:

   ```text
   /load build/ai/switch/edit-content.aider
   ```

3. сформулировать обычную смысловую задачу, просмотреть diff и запустить Draft;
4. если ИИ обоснованно сообщает о нехватке прав, выполнить предложенный им
   готовый переход, например:

   ```text
   /load build/ai/switch/edit-references.aider
   ```

Ввод `/load` пользователем является подтверждением перехода. Aider исполняет из
файла `/drop`, `/read-only` и `/add`; AutoNormoKontrol не управляет процессом
Aider и не выбирает capability по естественному языку. Канонический JSON и
правила режимов описаны в
[`docs/PROJECT_STRUCTURE.md`](PROJECT_STRUCTURE.md#capability-модель-ai-контекста).

Запускай Aider без произвольного стартового набора файлов: его `/drop` сохраняет
файлы, переданные как read-only при самом запуске процесса. Generated switch
гарантирует точность контекста, созданного AutoNormoKontrol, но не может отозвать
дополнительные полномочия, которые пользователь заранее выдал Aider вне проекта.

Если Draft позже выдаёт source suggestion, следует использовать его точный путь
как target нового плана. До этого нельзя угадывать главу по имени вроде
`content/02-main.md`.

После ответа браузерной модели её текст вставляется через `/paste`, затем
проверяется `/diff`. В официальной документации Aider также предусмотрен режим
`--copy-paste`, автоматизирующий локальные операции с буфером обмена; вставка в
сам браузерный сервис и копирование ответа остаются ручными. Формат `diff`
эффективен для локальных правок, потому что передаёт только блоки поиска и
замены, но конкретный edit format лучше оставлять выбранным Aider для модели.

Официальные справочные страницы:

- <https://aider.chat/docs/usage/copypaste.html>;
- <https://aider.chat/docs/usage/commands.html>;
- <https://aider.chat/docs/usage/modes.html>;
- <https://aider.chat/docs/more/edit-formats.html>.

## Насколько это удобно

| Режим | Удобство | Основной риск |
|---|---|---|
| Агент с доступом к репозиторию | Максимальное | Нужны права на локальные действия |
| Aider + браузерный чат | Среднее/высокое | Нужен локальный Aider/editor model и контроль diff |
| Чистый copy-paste | Среднее для малых правок | Потеря файлов, версии или части контекста |
| Передача полного файла после каждой правки | Низкое на больших главах | Дорого по токенам и легко затереть параллельные изменения |

Aider особенно полезен для повторяющегося цикла правок. Для одного небольшого
абзаца ручной diff проще. Для полного смыслового аудита ни один режим не
устраняет необходимость передать весь проверяемый снимок и привязать результат
к штатному `content-hash`.

## Обязательные меры безопасности

- Проверить правила и политику конфиденциальности выбранного веб-сервиса.
- Не загружать подписи, удостоверения и лишние персональные данные.
- Не считать текст «тесты пройдены» доказательством без локального вывода CLI.
- Не применять патч вслепую; просматривать diff до сборки.
- Не разрешать браузерному ИИ менять `external-acceptance.status` на `accepted`.
- Не переносить `semantic-review.status: pass` на изменённый снимок.
