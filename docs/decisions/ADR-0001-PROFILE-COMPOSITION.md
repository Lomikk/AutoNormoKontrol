# ADR-0001: независимые профили и отложенная композиция требований

Статус: принято для развития после однопрофильного MVP.

Дата: 16 июля 2026 года.

Область roadmap: R4 — второй профиль; R5 — композиция и overrides.

## Контекст

AutoNormoKontrol начинался как один проверяемый профиль курсовой работы ЮУрГУ.
После извлечения профиля из общего движка возник вопрос, нужно ли заранее
построить иерархию вида:

```text
СТО → университет → тип работы → кафедра → дисциплина → преподаватель
```

Такая схема обещает уменьшить дублирование, но для её достоверного построения
пришлось бы заранее исследовать множество стандартов и угадать будущие границы
применимости. Совпадение формулировки или значения в двух документах ещё не
означает, что существует общий нормативный родитель. Например, одинаковые
`Times New Roman 14 пт` в стандартах двух вузов остаются двумя требованиями с
разным происхождением, хотя технически их может реализовать один обработчик.

Текущий `profile-v1` описывает один полный профиль и намеренно не содержит
`extends`, `imports` или алгоритм merge. Это не считается дефектом MVP.

## Решение

### 1. Единица подключения — полный независимый профиль

Каждый поддерживаемый тип работы получает самостоятельный пакет, содержащий:

- точный набор применимых нормативных пунктов;
- provenance исходных стандартов и локальных решений;
- system prompt и inventories semantic/external review;
- обязательную структуру и metadata;
- шаблон, стили, фильтры и PDF-postflight;
- собственные fixtures и coverage/mutation tests.

Профиль не получает правила другого профиля неявно. До реализации resolver
профиль обязан быть самодостаточным, даже если его декларативные требования
частично повторяют соседний пакет.

### 2. Универсального родительского профиля «СТО» не существует

СТО — вид нормативного источника, а не единый стандарт всех документов и
университетов. Не создаётся пустой либо предположительный `sto-common` или
`university-common` только потому, что несколько значений выглядят одинаково.

Общий нормативный компонент допустим, только когда существует реальный источник
с доказанной областью действия сразу для нескольких профилей. В остальных
случаях сохраняются независимые requirement records с разным provenance.

### 3. Повторное использование начинается с механизмов

Дублирование классифицируется так:

| Повторяется | Решение |
|---|---|
| PowerShell/Lua/TeX-алгоритм | извлечь общий обработчик движка |
| Механика Draft/Strict, snapshot, coverage, diagnostics | оставить в engine |
| Значение настройки из разных нормативных источников | сохранить в обоих профилях |
| Один нормативный источник с общей областью действия | кандидат в нормативный компонент |
| Титульная форма или обязательная структура | вариант или отдельный профиль документа |
| Подтверждённая локальная хотелка | будущий явный override-компонент |

Таким образом, допустимо дублирование небольшого декларативного описания. Нельзя
копировать и независимо развивать одинаковые большие валидаторы, build scripts
или renderer implementations.

### 4. Общее извлекается после сравнения реальных профилей

Второй профиль сначала реализуется независимо. Затем для двух профилей строится
дифференциальная матрица:

- источник и точный пункт;
- область применимости;
- тип проверки: programmatic, semantic, external или informational;
- нормализованное правило и его параметры;
- используемый implementation key;
- различия metadata, структуры, форм и результатов postflight.

Извлечение разрешается, когда общность подтверждается этой матрицей. Для общего
обработчика достаточно одинаковой технической семантики; для общего
нормативного слоя дополнительно требуется общий источник или явно доказанная
область действия.

Первым архитектурным пробником рекомендуется реферат по официальному источнику,
а не ВКР и не временная статья: он достаточно отличается от курсовой, но не
вносит сразу максимальную сложность форм и внешних процедур.

### 5. Будущая модель — явная композиция, а не глубокое наследование

Если два независимых профиля докажут потребность, R5 может ввести effective
profile как детерминированную композицию:

```text
полный базовый профиль
+ явно выбранные versioned components
+ явно подтверждённые local overrides
= effective profile с digest и provenance
```

Список компонентов хранится явно в manifest или lock-файле workspace. Порядок
не выводится из имён каталогов. Конфликт одинакового приоритета завершается
fail-closed; молчаливое «последний файл победил» запрещено. Команда `explain`
должна показывать источник эффективного правила.

Это не фиксирует, сколько дополнений понадобится одной работе. Предпочтительная
единица публикации — один устойчивый контекст, например «преподаватель + тип
работы», внутри которого могут находиться несколько тематических правил. Список
компонентов не запрещён форматом будущего manifest, но его семантика не
реализуется до появления настоящего потребителя.

### 6. Другие университеты подключаются независимыми пространствами имён

Общий движок не должен знать ЮУрГУ, код кафедры или слово `coursework`.
Профили других организаций получают самостоятельные ID, нормативные корпусы и
тесты. Общими между вузами обычно будут engine primitives, а не нормативные
requirement records.

Формат физического каталога и способ распространения сторонних профилей
остаются решением R7. Эта неопределённость не должна блокировать R1 и R4.

### 7. Совместимость `profile-v1` не является самостоятельной целью

При появлении второго профиля допустима однократная миграция на `profile-v2`
без бессрочной поддержки обеих схем. Однако regression baseline текущего PDF,
диагностики, exit codes и трассировки сохраняется: он нужен для обнаружения
случайной поломки, а не для вечной поддержки старого manifest.

### 8. Backend не определяет нормативную архитектуру

Quarto не включается в MVP. Возможный эксперимент с ним рассматривается только
как замена части publication backend после стабилизации workflow. Профили,
requirements, provenance, Draft/Strict и external gates должны оставаться
независимыми от решения Pandoc напрямую или Pandoc через Quarto.

## Проверка по существующим проектам

Класс
[`BMSTU-IU8`](https://github.com/CatInCosmicSpace/latex-template/blob/master/tex/latex/bmstu-iu8/BMSTU-IU8.cls)
использует практичную неглубокую модель: варианты `diploma`, `research` и
`ordinary` выбирают отдельные формулировки и титульные функции, после чего класс
подключает общие модули рисунков, таблиц, формул, библиографии и приложений.
Пакет также опубликован на
[CTAN](https://ctan.org/tex-archive/macros/latex/contrib/bmstu-iu8/bmstu-iu8).

Из этого заимствуется принцип «общие устойчивые механизмы + явные варианты», но
не сама кодовая база: IU8-шаблон не реализует normative provenance,
semantic/external gates и AI-workspace AutoNormoKontrol.

## Последствия

Положительные:

- новый профиль нельзя случайно ослабить правилом соседнего профиля;
- происхождение каждого требования остаётся доказуемым;
- другие университеты не требуют универсального фиктивного стандарта;
- архитектура извлекается из реальных различий, а не из предположений;
- renderer и validators можно переиспользовать независимо от источника нормы.

Отрицательные:

- первые два профиля частично дублируют декларативные requirements и настройки;
- новый тип требует полного нормативного аудита, а не только смены титульника;
- общие компоненты появляются позднее, чем при заранее спроектированной
  иерархии.

Это принимаемый компромисс. Стоимость нескольких повторяющихся записей ниже
стоимости ошибочного общего слоя, который изменяет применимость требований.

## Отвергнутые варианты

1. Сначала исследовать все СТО и построить универсальное дерево — слишком
   дорого, не имеет ограниченного критерия завершения и задерживает полезный
   workflow.
2. Сделать `susu-common` копированием совпавших параметров — теряется различие
   между техническим совпадением и общей нормативной областью.
3. Глубокое наследование с неявным last-write-wins — конфликт может молча
   изменить Strict-профиль.
4. Форкнуть существующий TeX-шаблон как новый движок — не сохраняет gates,
   трассировку и agent contract AutoNormoKontrol.

## Триггер пересмотра

Решение пересматривается после полностью работающего второго профиля R4 либо
раньше, если реальный подтверждённый override невозможно представить отдельным
полным профилем без копирования существенного исполняемого кода.
