# ADR-0002: source inventory и profile requirements v2

Дата: 2026-07-19  
Статус: принято

## Контекст

В первой версии профиль вручную поддерживал `canonical-requirement-ids.json`,
`requirements.json`, `review-inventory.yaml` и два review template. Один пункт
повторялся в нескольких файлах, а диагностический текст оставался в Lua и
PowerShell. Добавление профиля требовало угадывать, какие копии считать
источниками истины.

## Решение

Оставить два нормативных источника истины:

- нейтральный `inventory.json` рядом с документом-источником;
- полный `requirements.json` внутри каждого профиля.

Движок компилирует из них exact review sets, журналы новой работы,
структурный граф, diagnostic catalog и build metadata. Исполняемые checks
выбираются только из фиксированного allow-list. Profile schema v1 и ручные
review templates удаляются; их обратная совместимость не является целью на
текущей стадии разработки.

## Последствия

- Нельзя сократить профильный реестр, не получив exact-set error относительно
  source inventory.
- Semantic/external записи больше не расходятся с требованиями.
- Пункты разных стандартов не смешиваются в одном «глобальном» реестре.
- Один пункт может иметь несколько независимых verification mechanisms.
- Старые обработчики мигрируют постепенно через явный `profile.handler`.
- Source inventory не определяет применимость и поэтому не является скрытым
  родительским профилем.
- SQLite не используется как источник истины: versioned JSON остаётся удобным
  для diff, code review, генерации и работы ИИ. Производный индекс можно добавить
  позднее без изменения контракта.
